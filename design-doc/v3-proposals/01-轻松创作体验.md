## 1. 背景与痛点

### 1.1 现状：创作路径长、认知负担重

当前用户从想法到第一章成文，需要走完：

```
CreateNovel（标题/简介/类型/基调/关联世界观）
  → Workbench/StoryConception（构思、角色卡）
  → OutlineWorkbench（逐章逐场景搭大纲）
  → ManuscriptWriter（逐场景触发生成）
  → 逐段修改（v2 总规划记录的基线：单章生成后需人工修改比例 ~60%）
```

痛点具体化：

1. **冷启动门槛高**：新用户面对空白的故事表单与大纲树不知从何下手；v2 用户画像中"爱好者"（周末写作、同人创作）的核心诉求正是"入门引导、灵感激发"。
2. **步骤间是断裂的**：构思、角色、世界观、大纲、正文分属不同页签，每一步都要用户主动发起，AI 不会"接着往下推"。
3. **逐场景手动触发生成**：写一章要多次点击-等待-检查；没有"整章/整卷草稿"的批量能力（v2 总规划已把"批量异步任务体系"列为欠账）。
4. **修改率高造成心理挫败**：~60% 的后期修改比例意味着"AI 写完之后用户的活才刚开始"，这是"不轻松"的最大来源——因此 G1 与 G2（初稿质量）互为表里。

### 1.2 业界参考

| 产品/方案 | 做法 | 可借鉴点 |
|-----------|------|----------|
| Sudowrite Story Engine | premise → synopsis → characters → outline → chapter beats → prose 的分步流水线，每步可改后再继续 | "全自动可跑通、每步可介入"的管道范式 |
| NovelAI | 极简：直接在编辑器续写 + lorebook 兜底 | 最低摩擦的"直接开写"模式 |
| 彩云小梦 | 一句话/一段开头即可续写，多分支候选供选 | 冷启动几乎为零 + 候选选择代替修改 |
| 各平台"作家助手"类产品 | 题材模板（系统流/穿越/重生等）预填设定 | 类型模板显著降低网文作者的搭建成本 |

## 2. 设计目标

| 编号 | 目标 | 度量 |
|------|------|------|
| G1-1 | 从想法到可读初章的时间显著缩短 | 新建故事→第一章完成的中位时长 -50% |
| G1-2 | 降低冷启动放弃率 | 新建故事后 24h 内产出正文的用户占比 +20pp |
| G1-3 | 初稿接受度提升（与 G2 协同） | 单章生成后人工修改比例 60% → <35%（v2 既定目标） |
| G1-4 | 不牺牲深度用户的控制权 | 现有 Workbench 全部入口保留，向导产物即标准数据（Story/Outline/Manuscript） |

## 3. 方案对比

| 维度 | 方案 A：引导式向导 + 一句话成稿管线 | 方案 B：对话式创作副驾 | 方案 C：类型模板库 + 智能默认 |
|------|------|------|------|
| 核心策略 | 结构化流水线：premise→设定→角色→大纲→分章草稿，全自动可跑通、每步可暂停介入 | 用户与"编辑 Agent"自然语言对话，Agent 调用现有 API 完成建档/大纲/生成 | 题材模板预填世界观/角色原型/大纲骨架，用户改空格即可开写 |
| 复杂度 | 中 | 高（Agent 循环 + 工具调用 + 前端会话界面） | 低 |
| 后端新增 | WorkflowOrchestrator + AsyncJobService + 2 张表 | Agent 运行时（工具调用协议、会话状态机） | 模板数据 + 少量 API |
| 前端改造 | 新增向导页 + 进度视图，Workbench 不动 | 新增对话面板（常驻侧栏），交互设计工作量大 | CreateNovel 页扩展 |
| 主要风险 | 流水线中某步质量差会级联放大 | 意图理解错误导致误操作数据；成本高 | 模板同质化恰与 G2 目标相悖 |
| 复用模块 | StoryService/OutlineService/ManuscriptService/PromptAssemblyService/SlopQualityGate | 同左 + AiService 工具调用 | Story/World/Outline 模板化 |
| 适用画像 | 爱好者、网文作者冷启动 | 全画像（长期形态） | 网文作者（题材套路明确） |
| 与其他目标协同 | 管线中的正文步直接挂 G2 竞技场 | Agent 可调用 G3 技法卡/G4 续写 | 模板可挂 G3 拆解出的结构模式 |

## 4. 方案 A 详设：引导式向导 + 一句话成稿管线（推荐首选）

> 当前实现边界（2026-07-14）：Step0-4、持久化后台任务、自动推进和标准实体落库已交付；Step5 正文草稿及 G2 竞技场接入尚未实施，以 `PENDING-PHASES.md` 为准。

### 4.1 用户体验流

```
入口："快速开书"（与现有"新建小说"并列）
Step0 一句话想法（必填，其余全部可选）
Step1 AI 扩写 premise（3 个候选方向，用户选/改/重掷）
Step2 核心设定 + 世界观骨架（可跳过；产物写入 World 草稿）
Step3 主要角色卡（AI 生成 3-5 张，用户增删改；产物写入 CharacterCard）
Step4 卷/章大纲（AI 生成，树形可编辑；产物写入 Outline）
Step5 分章草稿（异步批量生成前 N 章；每章走现有生成链 + SlopQualityGate）
完成页：进入 Workbench，一切产物均为标准实体，后续与现有流程无异
```

关键交互原则：

- **每一步"AI 先给可用的默认值，用户只做减法"**——这是"轻松"的本质：把"从零构造"变成"挑选与修剪"。
- **每一步产出 3 个差异化候选**（廉价的 G2 手段：在规划层就分叉，避免所有用户的 premise 都长一个样）。
- **任意步可"全自动到底"**：用户点"直接生成到第 3 章"，流水线静默跑完，靠 AsyncJobService 汇报进度。

### 4.2 后端设计

新组件 `WorkflowOrchestrator`（story/ 模块）：

```
WorkflowOrchestrator
├── startRun(userId, template, seedIdea) → workflowRunId
├── advance(runId, stepId, userEdits)     // 接受用户对上一步产物的修改，推进下一步
├── regenerateStep(runId, stepId, hint)   // 重掷（hint 支持"更黑暗一点"这类自然语言微调）
└── autoRunTo(runId, targetStep)          // 全自动推进（异步）
```

- 每步的提示词装配走 `PromptAssemblyService.assembleGuidedStep(step, context)`（见 00 号文档 3.1），上下文携带此前所有步骤的确认产物。
- 步骤产物**落库即标准实体**（Story/World/CharacterCard/Outline/Manuscript），流水线表只存引用与状态——保证向导与自由模式随时互通，不产生第二套数据。

数据模型（草案）：

```sql
CREATE TABLE creation_workflow_runs (
    id            CHAR(36)  NOT NULL,
    user_id       CHAR(36)  NOT NULL,
    story_id      CHAR(36)  NULL COMMENT 'Step1 后回填',
    template_key  VARCHAR(64) NOT NULL COMMENT '流水线模板：quick_book / short_story / fanfic ...',
    current_step  VARCHAR(32) NOT NULL,
    steps_json    LONGTEXT  NOT NULL COMMENT '各步状态/候选/用户选择',
    status        ENUM('running','waiting_user','completed','cancelled') NOT NULL,
    created_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);
```

批量分章草稿复用 `AsyncJobService`（00 号文档 §4），单章内部完全走现有 `ManuscriptService.generateForScene` 链路，天然带质量门禁。

### 4.3 Token/成本影响

- Step1-4 均为短输出（数百至两千字），成本可忽略；Step5 与现状单章成本相同 ×N 章。
- 批量生成前给出**预估报价**（章数 × 单章均值，数据来自 EconomyService 历史）并要求确认。

### 4.4 实施要点

- P0：WorkflowOrchestrator + run 表 + Step0-4（同步交互部分）
- P1：Step5 批量草稿（依赖 AsyncJobService）+ 完成页接入 Workbench
- P2：候选质量接入 G2 竞技场；流水线模板扩展（短篇/同人/续写入口）

## 5. 方案 B 详设：对话式创作副驾

### 5.1 形态

Workbench 常驻右侧"编辑"会话面板。用户说"帮我把第三章拆成两章，后半章加一场追逐戏"，Agent 规划工具调用（OutlineService.splitChapter → ManuscriptService.generateForScene）并在执行前展示计划待确认。

### 5.2 架构

```
前端会话面板 ⇄ AgentSessionController
                   └── AgentRuntime（意图解析 → 工具规划 → 逐步执行 → 汇报）
                          ├── 工具集 = 现有各 Service 的白名单方法（带 dry-run 描述）
                          └── 会话状态（当前故事/章节焦点、近期操作栈、可撤销点）
```

关键设计约束：

1. **写操作一律先出计划再执行**（防止意图误解直接污染数据）；执行后产生可撤销记录（依赖 v2/04 版本控制快照）。
2. 工具白名单从窄到宽灰度：先只开放"生成/改写"类只读+追加操作，稳定后再开放结构性操作（拆章、删场景）。
3. Agent 的系统提示词进 `PromptTemplateRegistry` 版本化管理。

### 5.3 评估

- 优点：长期形态最优，天花板最高；所有后续能力（G3 技法卡、G4 续写）都能以"对话入口"暴露，学习成本趋近于零。
- 缺点：工程量与风险显著高于 A（意图理解、多轮状态、误操作防护、前端交互设计）；且强依赖 v2/04 版本控制先落地作为"后悔药"。
- 结论：**作为 A 的第二阶段演进**，不建议首发。A 的流水线步骤天然就是 B 的工具集，先做 A 不浪费。

## 6. 方案 C 详设：类型模板库 + 智能默认

### 6.1 形态

CreateNovel 页新增"从模板开始"：系统流/无限流/重生复仇/穿书……每个模板携带：世界观骨架（World 草稿）、角色原型组（CharacterCard 草稿）、三幕大纲骨架（Outline 草稿）、推荐风格画像（StyleProfile 预设）。

### 6.2 评估

- 优点：成本最低（主要是内容运营工作 + 少量预填 API），对目标画像"网文作者"立即有感。
- 缺点：**与 G2 存在张力**——模板越好用，产出越同质。缓解手段：模板只提供结构骨架不提供文字表达；模板与 G3 打通后，可由用户自己拆解喜欢的作品生成"私人模板"，避免全站共用一套。
- 结论：作为 A 的廉价补充（A 的 Step2-4 本质上就能消费模板数据），不单独立项。

---

## 7. 推荐路径

1. **首发方案 A**：结构化流水线，复用度最高、风险可控，直接攻击"冷启动+步骤断裂"两大痛点。
2. 方案 C 以"流水线模板"形式并入 A（template_key 机制已预留），仅需内容运营投入。
3. 方案 B 作为 v3 后期演进目标，前置条件：A 稳定运行 + v2/04 版本控制落地。
4. G1 的成败一半取决于 G2：若初稿仍需 60% 修改，任何向导都只是"更快地到达痛苦"。故 06 号文档建议 **G2 先行或与 G1 并行**。
