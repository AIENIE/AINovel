## 1. 背景与痛点

### 1.1 现状

v2 对"借鉴他人作品"的支持停留在**风格维度**：`StyleAnalysisJob` 可以对一段文本提取 8 维风格指纹并生成 StyleProfile（v2/02 的 G4"风格迁移"目标）。这解决了"文字听起来像"，但没有触及一部好作品真正值得学的东西：

- **结构层**：开篇钩子怎么埋、主线支线怎么编织、高潮怎么铺垫；
- **节奏层**：多少字一个爽点、压制-爆发周期怎么排布、章末悬念的类型分布；
- **人物层**：人物弧线的转折点设计、关系网的张力来源、反派的立体化手法；
- **技法层**：对话潜台词、信息差的制造与释放、伏笔的埋设与回收间距。

用户当前想学某部书的优点，只能自己读、自己悟、自己在提示词里手写"我想要类似 XX 的开篇"——门槛高且效果不可控。

### 1.2 合法性基础："吸取技法"而非"复制文本"

本目标的产品红线由著作权法的**思想/表达二分法**（idea/expression dichotomy）划定（详见 05 号文档 §4）：

- **不受保护、可自由吸收**：情节类型与结构（三幕、英雄之旅、压制-爆发）、节奏模式、人物原型、题材惯用场景（scènes à faire：升级流的副本、豪门文的宴会）；
- **受保护、绝不迁移**：具体文字表达、独有对话、高度个性化的角色刻画与专有设定。

因此系统的一切产物必须是**结构化的技法描述**（"该作品在第 3 章末用'身份揭穿倒计时'型悬念，间隔 2 章回收"），而非原文片段的搬运。生成端另设相似度防线兜底（§4.5）。

用户已确认：全文私有导入可接受（仅用于用户私人分析与创作辅助，不对外分发原文）。`reference_works` 表仍保留版权确认标记与用途声明。

### 1.3 研究支撑（详见 05 号文档）

| 来源 | 结论 | 对本方案的输入 |
|------|------|----------------|
| EIPE-text（2023） | 从叙事语料库迭代抽取"计划"构建计划库，用于指导长篇生成，连贯性与相关性显著提升 | 方案 B"技法卡库"的直接学术原型 |
| Dramatron / DOC | 层级分解（log line → 情节点 → 场景）是结构分析的天然骨架 | 拆解报告的分层维度设计 |
| 网文工艺研究 | 头部作品 1.8 章/情绪高潮、三层钩子体系、W/M/N 型情绪曲线可统计 | 节奏层拆解的量化指标 |
| WebNovelBench（2025） | 网文分布上的 LLM 评估已有基准方法 | 拆解质量的对照校准 |

## 2. 设计目标

| 编号 | 目标 | 度量 |
|------|------|------|
| G3-1 | 用户可导入参考作品并获得可读的深度拆解 | 导入→查看报告转化率；报告有用性评分 ≥ 7.5/10 |
| G3-2 | 拆解产物可直接作用于生成 | 技法卡启用后的产物在对应维度盲测优于未启用（如"钩子有效性"） |
| G3-3 | 零文本抄袭 | 生成文本与参考原文的 n-gram/向量相似度告警率 < 1%，命中即拦截重写 |
| G3-4 | 拆解可信 | 抽样人工校验：结构/节奏结论与人工标注一致率 ≥ 80% |

## 3. 方案对比

| 维度 | 方案 A：拆解报告 + 手动应用 | 方案 B：技法卡库 + 生成时注入 | 方案 C：多作品对比融合 |
|------|------|------|------|
| 核心策略 | 导入 → 分层拆解 → 输出人类可读的"写作技法报告"，用户自行消化运用 | 拆解产物结构化为可勾选的"技法卡"，直接注入生成管线 | 对 N 部参考作品交叉分析，提炼共性优点与差异，生成融合技法画像 |
| 用户价值 | 学习工具（授人以渔） | 创作增强（拿来即用） | 个人流派构建（博采众长） |
| 复杂度 | 中 | 中高（建立在 A 上） | 高（建立在 B 上） |
| 新增组件 | ReferenceWorkAnalyzer + 摄入管线 + 报告 UI | + technique_cards 表 + TechniqueCardInjector + 相似度防线 | + 跨作品对比分析任务 + 融合画像 |
| 复用模块 | Material RAG（分块/向量）、AsyncJobService、StyleAnalysisJob | + PromptAssemblyService（assembleWithTechniques）、G2 竞技场 | 同左 |
| 风险 | 报告"正确但无用"（太泛） | 技法卡滥用导致新同质化；相似度防线成本 | 融合结论玄学化、不可验证 |

**三方案为递进关系：A 是 B 的产出物之一（报告即技法卡的可读视图），C 是 B 的多作品扩展。核心投资是"摄入 + 拆解"管线，三方案共用。**

## 4. 方案 A+B 详设（合并叙述：B 包含 A 的全部产出）

### 4.1 摄入管线（与 G4 共建，见 04 号文档 §4.2）

```
上传参考作品（txt/epub，或粘贴）
  → 版权用途确认（私人分析用途声明，落库标记）
  → 分章解析（复用 MaterialChunker 思路，按章边界而非固定长度分块）
  → 三路并行处理（AsyncJobService 长任务，进度可见）：
     ① 向量化入 Qdrant（独立 collection：reference_works，与素材库隔离——
        用途不同：素材库供正文引用，参考库仅供分析与相似度检查）
     ② 逐章结构标注（LLM）：本章事件链 / 出场人物 / 钩子类型与位置 / 情绪拍点
     ③ 全书统计特征（本地计算）：章长分布、对话占比、句长节奏、高频意象
```

### 4.2 分层拆解（ReferenceWorkAnalyzer）

在逐章标注之上做四层聚合分析，每层产出报告章节 + 结构化技法卡：

| 层 | 分析内容 | 技法卡示例 |
|----|----------|-----------|
| 结构层 | 幕结构识别、主线/支线编织图、开篇模式（前 3 章冲突位置与信息布局）、高潮铺垫链 | 「开篇卡：首章 300 字内出现冲突；金手指在第 2 章以代价形式出现」 |
| 节奏层 | 情绪曲线拟合（W/M/N 型）、爽点间距统计、章末钩子类型分布、压制-爆发周期 | 「节奏卡：平均 2.1 章一次爆发；爆发前必有 1 章蓄力；章末 70% 用事件钩」 |
| 人物层 | 主要人物弧线转折点、关系网张力矩阵、人物语言区分度 | 「弧线卡：主角每次能力提升伴随一次关系代价」 |
| 技法层 | 信息差的制造/持有/释放模式、伏笔埋设-回收间距、对话潜台词密度、描写取景习惯 | 「伏笔卡：平均埋设后 8-15 章回收；回收时点在情绪低谷» |

拆解提示词的证据规范复用 slop 诊断的既有约束：**每个结论必须引用章节位置证据**，禁止空泛断言（保证 G3-4 可校验）。

### 4.3 技法卡数据模型（草案）

```sql
CREATE TABLE reference_works (
    id            CHAR(36) NOT NULL,
    user_id       CHAR(36) NOT NULL,
    title         VARCHAR(200) NOT NULL,
    source_note   VARCHAR(500) NULL COMMENT '来源与用途声明',
    rights_ack    TINYINT(1) NOT NULL DEFAULT 0 COMMENT '用户已确认私人分析用途',
    chapter_count INT NOT NULL,
    ingest_status ENUM('pending','processing','ready','failed') NOT NULL,
    stats_json    LONGTEXT NULL COMMENT '全书统计特征',
    PRIMARY KEY (id)
);

CREATE TABLE technique_cards (
    id           CHAR(36) NOT NULL,
    work_id      CHAR(36) NULL COMMENT '来源作品；NULL 为系统预置/融合卡',
    user_id      CHAR(36) NOT NULL,
    layer        ENUM('structure','pacing','character','craft') NOT NULL,
    title        VARCHAR(200) NOT NULL,
    description  TEXT NOT NULL COMMENT '技法的抽象描述（不含原文）',
    evidence_json TEXT NULL COMMENT '章节位置级证据引用',
    injection_prompt TEXT NOT NULL COMMENT '注入生成时的指令化表述',
    scope        ENUM('opening','scene','chapter_end','arc','global') NOT NULL,
    enabled_stories_json TEXT NULL,
    PRIMARY KEY (id)
);
```

`work_analysis_reports` 表存放报告全文（分层 JSON + 渲染 markdown），报告即技法卡的可读聚合视图。

### 4.4 生成时注入（TechniqueCardInjector）

- 用户在故事设置中勾选启用的技法卡（按 scope 自动匹配场景：开篇卡只在前 3 章生效、章末卡只影响结尾段落）；
- `PromptAssemblyService.assembleWithTechniques()` 将卡片的 `injection_prompt` 作为**结构性指令**注入（位于风格画像之后、负面清单之前，优先级见 00 号文档 §3.1）；
- 与 G2 管线的融合点：技法卡优先作用于**节拍规划阶段**（结构/节奏卡影响节拍方案），仅 craft 类卡作用于起草阶段——这符合"结构学骨、不学皮"的原则；
- 防新同质化：单场景注入的技法卡数量上限 3 张；系统提示用户"卡是参考不是配方"。

### 4.5 相似度防线（防抄袭兜底）

生成完成后、SlopQualityGate 之前增加一道 `SimilarityGuard`：

1. **n-gram 精确匹配**：生成文本与该用户启用的参考作品做 12-gram 以上重叠检测（本地，零成本）；
2. **向量相似**：对生成文本分块查询 reference_works collection，段级余弦相似度超阈值（建议 0.92 起步，实测校准）即告警;
3. 命中处理：标记该段并自动触发一次"换表达重写"（复用保守修订服务，指令为"保留情节、彻底更换表达"）；二次命中则拦截并提示用户。

### 4.6 成本

- 摄入 + 拆解为一次性长任务：约全文 token 的 1.2~1.5 倍（逐章标注为主），百万字作品约数十分钟（异步）；拆解前给出预估报价并确认。
- 技法卡注入的边际成本≈0（数百 token 的提示词增量）；相似度防线本地为主，向量查询成本可忽略。

## 5. 方案 C 详设：多作品对比融合

### 5.1 形态

用户选 2-5 部已拆解的参考作品，发起对比分析任务：

- **共性提炼**：多部作品在同层反复出现的技法 → 高置信"流派共性卡"（如三部都满足"爆发前必蓄力"）；
- **差异图谱**：同一维度的不同选择并列呈现（A 书快节奏高密度 vs B 书慢热深铺垫），让用户理解 trade-off 而非只见招式；
- **融合画像**：用户按喜好从各书勾选维度，合成个人"技法画像"（一组技法卡 + 一份 StyleProfile 建议），作为新书的默认配置——这是 G1 方案 C"私人模板"的高级来源。

### 5.2 风险控制

- 融合结论必须保留证据链（每条共性卡列出各书的证据位置），否则退化为玄学；
- 建立在 B 的卡片体系上，无新增基础设施，主要是分析提示词与 UI 工作。

## 6. 推荐路径

1. **一期做 A+B 合体**（摄入管线 + 四层拆解 + 报告 + 技法卡注入 + 相似度防线）：A 单独做价值不闭环（报告看完难落地），B 的增量成本主要是注入器与防线，合并交付性价比最高。
2. 摄入管线**与 G4 同期共建**（见 04 号文档）：两者的分章解析、向量化、逐章标注完全同构，G4 只是多了指纹提取与状态库构建。
3. 方案 C 二期跟进，依赖 ≥ 一批用户完成多作品拆解后的真实需求验证。
4. 预置卡片库：用系统对若干公版/授权作品预拆解，提供开箱即用的技法卡集，解决冷启动。
