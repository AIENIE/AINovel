## 1. 背景与需求定义

### 1.1 场景

用户拿到一部停止更新的小说（太监文，10 万~300 万字皆有可能；可能是别人的书，也可能是自己弃坑的旧作），希望系统续写，要求：

1. **像**：读起来是"原作者回来了"——文风、叙事习惯、人物声音、节奏气质都对；
2. **对**：不违背已有事实——人物设定、世界规则、时间线、伏笔、断点时刻的人物状态；
3. **活**：不被原文锁死——剧情要有新的推进和惊喜，而不是原有桥段的复读与平庸的惯性延伸。

三者存在内在张力：**"像"与"对"都是收敛力，"活"是发散力**。本文档的核心设计就是让这组张力显式化、可配置、可监控，而不是听天由命。

### 1.2 社区实践的失败模式（本方案的反面清单，详见 05 号文档 §5）

| 失败模式 | 根因 | 本方案的对策 |
|----------|------|--------------|
| 人物 OOC（前俏皮后沉稳） | 无全文级人物状态与声音约束 | 状态库 + CharacterVoice 提取 + 弧线方向标注（§4.3） |
| 伏笔遗忘 / 道具消失 | 无显式伏笔与资源账本 | 伏笔清单 + 世界状态快照（§4.3） |
| 风格漂移（越写越不像，长篇尤甚） | 风格只靠开头几段提示，随距离衰减 | 指纹持续注入 + 例句轮换 + 漂移监控闭环（§4.4/§5.2） |
| 惯性平推（像但无聊） | 模型沿最高概率路径延伸旧模式 | 走向候选发散 + 自由度契约（§5.1） |
| 直接塞全文进长上下文 | "中间丢失"：长上下文中部信息利用率暴跌 | 摘要分层 + RAG 检索式装配，不裸塞全文（§4.5） |

### 1.3 研究支撑（详见 05 号文档）

- 少样例/续写式提示对风格匹配有数量级提升（few-shot 23.5×；completion 式提示风格一致性最高）→ **例句集与"接着写"式装配**是性价比最高的像化手段；
- 作者风格嵌入（LUAR/StyleDistance 类）与作者语言模型困惑度可量化"像不像"→ 保真度可以做成**自动指标**而非纯人审；
- 社区实测：以原作前若干章为仿写范本的续写，风格稳定性显著优于纯指令描述 → 指纹中**例句集权重高于形容词描述**；
- GRPO 微调可保持长篇作者风格一致性 → 方案 C 的技术依据。

## 2. 设计目标

| 编号 | 目标 | 度量（详见 06 号文档） |
|------|------|------|
| G4-1 | 风格保真 | 盲测：读者区分"原作段 vs 续写段"的正确率接近随机（≤ 60%）；风格距离指标（嵌入余弦/困惑度）落入原作各章的自身波动区间 |
| G4-2 | 事实一致 | 续写内容与状态库的矛盾 issue 率 < 5%（复用连续性检查思路） |
| G4-3 | 剧情鲜活 | "创意自由度"人工评分 ≥ 7/10；走向候选与原文既有桥段的语义重复率受控 |
| G4-4 | 伏笔治理 | 用户标记为"须回收"的伏笔 100% 进入规划视野（回收/显式搁置二选一，不允许静默遗忘） |
| G4-5 | 零文本抄袭 | 与原作的表达级相似度防线（同 G3 §4.5） |

## 3. 方案对比

| 维度 | 方案 A：摄入 + 指纹 + RAG 续写 | 方案 B：A + 续写规划器 + 自由度契约 + 保真度闭环 | 方案 C：B + 单作品微调 |
|------|------|------|------|
| 核心策略 | 把续写做成"超强上下文装配"问题：指纹 + 状态库 + 检索前文 + 原文尾窗 | 在 A 之上把"像/对/活"的张力产品化：走向候选、契约分区、漂移监控 | 对超长原作训练 per-work LoRA，从权重层面获得"底味"，B 的管线不变 |
| 解决"像" | 指纹 + 例句 + 尾窗仿写 | 同左 + 漂移监控自动纠偏 | 权重级风格内化（上限最高） |
| 解决"对" | 状态库注入 | + 生成后连续性核查闭环 | 同 B |
| 解决"活" | 弱（依赖 G2 管线的发散） | 强（走向候选 + 契约显式划分自由区） | 同 B |
| 模型要求 | M0 可行 | M0 可行（M1 更佳） | M2 |
| 复杂度 | 中高 | 高 | 很高 |
| 单作品启动成本 | 摄入一次性成本（全文 1.2~1.5× token） | 同左 + 规划交互 | + 训练成本（数百万字语料的 LoRA 训练） |
| 适用 | 快速可用版 | 产品完全体（推荐） | 超长原作（>100 万字）或商业级保真需求 |

**A 是 B 的子集，B 是推荐形态；C 是 B 在特定场景下的增强插件而非替代。以下按 B 为主线详设，A 即 B 去掉 §5 的产物。**

## 4. 方案 B 详设（一）：摄入与资产构建

### 4.1 总体流程

```
导入原作全文（版权确认，同 G3）
  → 摄入管线（与 G3 共建：分章、向量化、逐章标注）
  → 三大资产并行构建（AsyncJobService，百万字级约数十分钟）：
     ① 作者指纹（AuthorFingerprintBuilder）
     ② 故事状态库（实体/关系/时间线/伏笔/断点快照）
     ③ 分层摘要树（全书→卷→章三级，供检索装配）
  → 用户校对关键资产（人物卡、伏笔清单、断点状态——人工确认一次，后续全自动）
  → 进入续写会话（continuation_sessions）
```

用户校对环节是刻意设计：状态提取的错误在续写中会指数放大，一次 10 分钟的人工确认远优于事后修
补；同时这也是用户建立信任感的关键触点。

### 4.2 作者指纹（AuthorFingerprintBuilder，style/ 模块）

指纹 = 现有 StyleProfile 8 维画像的**强化超集**，四个组成部分按注入优先级排列：

| 组件 | 内容 | 构建方式 | 注入方式 |
|------|------|----------|----------|
| 例句集（最高权重） | 各场景类型（动作/对话/心理/过渡/章末）的代表性原文段落各 3-5 段 | LLM 按"最能代表作者习惯"标准全书筛选 + 统计特征校验 | 每次续写按当前场景类型抽 2-3 段作仿写范本，**轮换采样**防止例句本身被复读 |
| 统计特征 | 句长分布与节奏、对话占比、段落长度、虚词习惯、标点癖好、高频私有词 | 本地计算（零成本、客观） | 转译为可执行指令（"平均句长 18 字，短句收尾偏好"） |
| 8 维画像 + 叙事习惯描述 | 复用 StyleAnalysisJob，增补：视角习惯、转场方式、描写取景、幽默机制 | LLM 分析（分章采样聚合） | 系统提示词（现有 StyleContextProvider 通道） |
| 禁忌集 | 作者明显**不做**的事（如从不写内心独白、从不用感叹号） | 统计 + LLM 联合判定 | 负面清单（与 G2 负面约束合流） |

角色声音：对主要人物各建 CharacterVoice（复用现有实体），从原作对话中提取口癖、句式、情绪反应模式与 3-5 段样本对话。

### 4.3 故事状态库（"七账本"）

综合社区实践总结与 v2/01 设计，状态库包含：

| 账本 | 内容 | 载体 |
|------|------|------|
| 人物卡 + 弧线位置 | 设定、能力、当前目标；弧线处于哪个阶段、下一步方向 | CharacterCard 扩展 + KAG |
| 关系矩阵 | 两两关系的当前状态与最近变化方向 | KAG 边属性 |
| 世界规则 | 力量体系、社会规则、已确立的"物理定律" | Lorebook 条目 |
| 资源账本 | 道具、金钱、境界等级等可数状态 | Lorebook 条目（结构化） |
| 时间线 | 主要事件的先后与间隔 | KAG 事件链 |
| 伏笔清单 | 每条伏笔：埋设位置、内容、预期方向、状态（未回收/已回收/显式搁置） | 新表 foreshadowing_items |
| 断点快照 | 停更时刻：谁在哪、正在干什么、悬而未决的即时冲突 | continuation_sessions 内 JSON |

依赖说明：KAG（Neo4j）完整构建是 v2/01 的未完成项，本目标是其最强需求方；若 KAG 未就绪，账本可先全部落 Lorebook + JSON（功能等价、检索稍弱），**不阻塞**。

### 4.4 分层摘要树与检索装配（对策"中间丢失"）

续写单章的上下文装配（`PromptAssemblyService.assembleContinuation()`），预算内优先级从高到低：

1. 续写契约与本章节拍计划（硬约束，见 §5.1）；
2. 状态库相关切片：本章出场实体的账本条目（按节拍计划中的实体检索）；
3. 作者指纹（例句按场景类型轮换）；
4. **原文尾窗**：断点前 1-2 章原文全文（"接着写"的直接语感来源，completion 式衔接）；
5. 检索前文：按本章相关实体/伏笔从向量库检索的关键原文段（含每段章节定位）；
6. 摘要树：全书一段 + 相关卷/章摘要。

明确不做：将全文塞入长上下文。百万字中文远超可用窗口，且长上下文中部信息利用率显著退化（lost in the middle），检索式装配在成本与效果上双优。

## 5. 方案 B 详设（二）："活"的机制——规划器与契约

### 5.1 续写规划器（NovelContinuationPlanner）与自由度契约

**续写开始前的一次性规划交互：**

```
① 系统基于状态库生成"续写形势报告"：
   未回收伏笔清单 / 各人物弧线的悬空状态 / 断点冲突 / 原作已显露的走向暗示
② 生成 3-5 条差异化的中期走向候选（每条 = 未来 10-20 章的故事弧概要），
   显式标注：回收哪些伏笔、推进哪些弧线、引入什么新变量、风险点是什么
   （发散引擎复用 G2 节拍规划的 Verbalized Sampling 范式，走向之间必须策略性不同）
③ 用户选择/混合/修改走向 → 形成"续写契约"（continuation_sessions.contract_json）
```

**自由度契约的结构**——把"像/对/活"的张力从隐式变成显式配置：

| 分区 | 内容 | 效力 |
|------|------|------|
| 必守区（hard） | 世界规则、人物核心设定、已确立事实、用户勾选"必须回收"的伏笔 | 硬约束注入 + 生成后核查，违反即重写 |
| 引导区（soft） | 选定走向、弧线方向、节奏基调 | 注入节拍规划，可被更好的局部创意突破（突破时要求模型自述理由，供用户裁决） |
| 自由区（free） | 新人物、新支线、新场景的具体设计；表达层的一切 | 不约束，交给 G2 发散管线 |
| 显式放弃区 | 用户标记"可以烂尾/不再理会"的原作线头 | 从规划视野移除，防止系统为全部旧线头所累 |

**自由度滑杆**（fidelity slider，0-100）：用户级参数，控制引导区的松紧与走向候选的发散半径。低值≈"忠实同人"，高值≈"精神续作"。这是对"不会过于被已有的内容限制住"这一需求的直接产品化。

### 5.2 保真度监控闭环（SlopDriftService 反向复用）

现有 `SlopDriftService` 做"前文 vs 后文是否变 AI"的多窗口比较；续写场景将其调转方向：

```
每完成 N 章续写（默认 5）自动运行 FidelityMonitor：
  维度1 风格距离：续写窗口 vs 原作代表窗口
        - 本地统计特征距离（句长/对话占比/虚词分布——零成本、客观）
        - 风格嵌入余弦距离（若引入作者嵌入模型）或 LLM 评审（降级方案）
        - 判定基线：原作各章之间的自身波动区间（作者本人也有波动，
          超出该区间才算漂移——避免把正常创作弹性误报为失真）
  维度2 事实一致：续写内容 vs 状态库（复用 v2/03 连续性检查设计）
  维度3 惯性复读：续写章 vs 原作既有桥段的语义相似度（防"换皮重演"）
  产出：保真度报告（continuation_fidelity_runs），超阈值 → 提醒用户 +
        给出定位证据 + 一键"校正重写该章"（携带漂移诊断的修正指令）
```

监控是**闭环的最后一块**：指纹注入管"写之前像"，漂移监控管"写了十万字之后还像"——这正是社区实践中最常见的崩坏点（现有研究记录的"十万字后 AI 接管"现象的续写版）。

### 5.3 单章续写管线（整合视图）

```
契约 + 形势 → 节拍规划（G2 管线，携带引导区约束与伏笔调度）
           → 带约束起草（指纹 + 尾窗 + 检索前文 + G2 负面清单；N 候选）
           → 竞技场选优（评分维度追加"风格保真分"）
           → SimilarityGuard（对原作，同 G3 §4.5）
           → SlopQualityGate（现有门禁不变）
           → 状态库增量更新（新事实/新伏笔/弧线推进写回账本）★
           → 每 N 章触发 FidelityMonitor
```

★ 状态库写回是长篇不崩的关键：续写产生的新事实与原作事实同等入库，后续章节据此装配——系统自身的记忆闭环。

### 5.4 数据模型（草案）

```sql
CREATE TABLE continuation_sessions (
    id             CHAR(36) NOT NULL,
    user_id        CHAR(36) NOT NULL,
    work_id        CHAR(36) NOT NULL COMMENT '引用 reference_works（共用 G3 摄入）',
    story_id       CHAR(36) NOT NULL COMMENT '续写内容挂载的本站故事实体',
    fingerprint_id CHAR(36) NOT NULL,
    contract_json  LONGTEXT NOT NULL COMMENT '自由度契约：分区 + 滑杆值 + 选定走向',
    breakpoint_json LONGTEXT NOT NULL COMMENT '断点快照',
    status         ENUM('planning','writing','paused','completed') NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE author_fingerprints (
    id            CHAR(36) NOT NULL,
    work_id       CHAR(36) NOT NULL,
    profile_json  LONGTEXT NOT NULL COMMENT '8维画像+叙事习惯',
    exemplars_json LONGTEXT NOT NULL COMMENT '分场景例句集',
    stats_json    LONGTEXT NOT NULL COMMENT '统计特征',
    taboos_json   TEXT NULL COMMENT '禁忌集',
    PRIMARY KEY (id)
);

CREATE TABLE foreshadowing_items (
    id          CHAR(36) NOT NULL,
    session_id  CHAR(36) NOT NULL,
    planted_at  VARCHAR(64) NOT NULL COMMENT '埋设章节定位',
    summary     TEXT NOT NULL,
    status      ENUM('open','resolved','abandoned') NOT NULL DEFAULT 'open',
    must_resolve TINYINT(1) NOT NULL DEFAULT 0 COMMENT '用户勾选必须回收',
    resolved_at VARCHAR(64) NULL,
    PRIMARY KEY (id)
);
-- continuation_fidelity_runs 结构衍生自 slop_drift_runs，增加对照基线字段
```

续写产物挂载为标准 Story/Outline/Manuscript 实体：Workbench 全部现有能力（编辑、版本、质量诊断）对续写内容天然可用。

## 6. 方案 C 详设：单作品微调轨道

### 6.1 定位

方案 B 的指纹注入存在天花板：提示词层面的模仿对**隐性风格特征**（无意识的语言习惯）复现有限（见 05 号文档 §2.1）。对超长原作（>100 万字，语料充足）且用户对保真有极高要求的场景，训练 per-work LoRA 作为"底味"层：

- 训练语料：原作全文（该会话用户私有；LoRA 权重同样私有，不跨用户共享）；
- 训练目标：续写式语言建模为主，可选 GRPO 强化长篇风格一致性；
- 推理：LoRA 提供底味，方案 B 的契约/状态库/监控管线原样保留（**微调解决"像"，解决不了"对"和"活"**——三者机制独立，这是把 C 定位为插件而非替代的原因）。

### 6.2 门槛与风险

- 依赖 M2：可微调底座（开源权重模型 + 训练设施，或外部微调 API）——与 G2 方案 C 共享此决策与管线；
- 单作品训练成本需产品化定价（预估报价 + 确认，参照摄入的模式）；
- 版权敏感度高于分析场景（在用户私有、不分发权重的边界内运行；详见 05 号文档 §4 的地区差异），需在服务条款中明确用途限制与用户责任。

## 7. 推荐路径

1. **按 B 的完全体立项，分两期交付**：
   - 一期 = 方案 A 范围（摄入 + 指纹 + 状态库 + RAG 续写 + SimilarityGuard），已可用且明显优于市面"贴前文续写"类产品；
   - 二期 = 规划器 + 自由度契约 + FidelityMonitor（"活"与"长期不崩"的完全体）。
2. 摄入管线与 G3 同期共建；G2 管线是起草质量的前置依赖——因此 G4 排期在两者之后（见 06 号文档总排序）。
3. 方案 C 与 G2 方案 C 合并决策（同一套 M2 基础设施），触发条件：B 上线后保真盲测仍不达标 G4-1，或出现明确付费意愿的高保真需求。
4. 冷启动运营建议：用公版名著或授权作品做公开演示案例（规避演示环节的版权争议）。
