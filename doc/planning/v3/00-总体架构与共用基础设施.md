## 1. 定位与设计原则

v3 提案不是对当前实现的重构，而是在既有基础设施之上，补齐四项**能力型目标**：

| 编号 | 目标 | 对应文档 |
|------|------|----------|
| G1 | 让用户更轻松地用 AI 创作小说 | `01-轻松创作体验.md` |
| G2 | 生成端反 AI 味：更有创造力、更亲和人类兴趣、更吸引人 | `02-生成端反AI味与创造力.md` |
| G3 | 便捷地从其他小说著作中吸取优点，融入自己的小说 | `03-参考作品拆解与优点吸收.md` |
| G4 | 对停更小说续写：模拟原作者感觉，且不被已有内容过度限制 | `04-停更小说续写.md` |

四个目标遵循统一的设计原则：

1. **复用优先**：已交付的风格系统、素材 RAG、反 slop 质量门禁和 Lorebook/图谱能力是 v3 的地基。任何方案必须先说明"复用什么"，再说明"新增什么"。
2. **生成端优先于检测端**：现有 `quality/` 模块、`doc/architecture/quality.md` 和研究汇总已形成 E1-E4 证据等级、模式库与多窗口漂移检测。v3 的重心是把这套**检测知识反向输入生成过程**——从“写完了再挑毛病”升级为“落笔时就避开毛病”。
3. **可度量**：每个目标必须绑定可观测指标（盲测得票率、slop 风险分中位数、初稿接受率、风格距离），基线数据来自现有 `slop_quality_runs` / `slop_drift_runs` 遥测表。
4. **方案可选**：每个目标提供 2-3 个复杂度/成本/风险不同的方案（A 最轻、C 最重），允许按预算与优先级独立取舍；方案之间尽量共享本文档定义的公共组件。
5. **模型策略开放**：经决策确认，v3 方案允许突破当前 `AiModelPolicy` 对 `deepseek-v4-flash` 的锁定，包含多模型路由与微调/LoRA 轨道；但每个目标的方案 A 必须在锁定模型下可行，作为保底路径。

## 3. 共用扩展点

### 3.1 PromptAssemblyService：装配方法族

现有 `assembleSceneDraft()` 保持不变，新增并列的装配入口（各目标文档给出具体输入结构）：

```
PromptAssemblyService
├── assembleSceneDraft(...)                 [现有] 场景草稿
├── assembleBeatPlan(...)                   [G2] 节拍规划（先发散剧情节拍，再写正文）
├── assembleWithCreativeConstraints(...)    [G2] 携带负面模式约束 + 人味要素清单的正文装配
├── assembleGuidedStep(...)                 [G1] 向导流水线各步骤（premise/大纲/角色）的装配
├── assembleWithTechniques(...)             [G3] 注入所选"技法卡"的正文装配
└── assembleContinuation(...)               [G4] 续写装配（作者指纹 + 续写契约 + 检索前文）
```

设计要点：

- 全部走统一的 Token 预算归一化逻辑（现有 128K 默认 / 256K 上限），新增上下文按优先级裁剪：**硬约束（契约/事实） > 风格指纹 > 技法卡 > 检索片段 > 负面模式清单**。
- 负面模式清单（来自 slop 模式库）注入时做**采样轮换**而非全量注入：每次生成随机抽取 15-25 条高权重模式，避免提示词本身固化出新的模板味，也控制 token 开销。

### 3.2 StyleContextProvider：注入器扩展

```
StyleContextProvider
├── 风格画像注入                      [现有]
├── 角色声音注入                      [现有]
├── +CreativePersonaInjector          [G2-B] 创意人格（叙事者气质，与风格画像正交）
├── +TechniqueCardInjector            [G3] 参考作品技法卡
└── +AuthorFingerprintInjector        [G4] 作者指纹（8 维画像的强化版 + 例句集 + 禁忌集）
```

### 3.3 SlopQualityGate：从"守门员"到"评委"

现有门禁是**串行末端**（生成 → 评估 → 保守修订）。v3 新增两种调用形态，不改变现有链路：

1. **竞技场评委（G2）**：对 N 个候选并行打分（复用 `LocalSlopHeuristics` 快速档，必要时升级 `SlopDiagnosticService` 精细档），输出"slop 风险分 + 创意/人味加分项"的复合分，供选优。
2. **保真度监控（G4）**：`SlopDriftService` 的多窗口比较逻辑反向复用——不再比较"前文 vs 后文是否变 AI"，而是比较"续写 vs 原作窗口"的风格距离，超阈值触发提醒或重写。

### 3.4 MaterialRetrievalService：参考作品检索

现有素材混合检索（关键词 + 向量）直接复用；G3/G4 新增两类检索意图：

- `retrieveReferenceTechniques(workId, intent)`：按"当前要写的场景类型"检索参考作品的技法卡与对应例段。
- `retrieveSourceContext(workId, entities, window)`：续写时按出场实体检索原作关键前文（伏笔、关系变化节点）。

## 6. 模型策略

### 6.1 三档模型形态

| 档位 | 形态 | 适用 | 前提 |
|------|------|------|------|
| M0 | 现状：`deepseek-v4-flash` 单模型 | 各目标方案 A 的保底路径 | 无 |
| M1 | 多模型路由（能力标签、任务路由与回退链） | 规划/起草/润色/评审分工；创意任务用高发散模型，评审用严谨模型 | 放开 `AiModelPolicy` 锁定；AI 网关侧支持多模型计费 |
| M2 | 微调/偏好优化轨道（风格 LoRA、基于 slop 标注的 DPO） | G2 方案 C、G4 方案 C | 训练基础设施或外部微调 API；数据管线（`candidate_items` 沉淀的偏好对） |

### 6.2 解锁 `AiModelPolicy` 的建议路径

1. 先在 **评审/分析类任务**（slop 判定、拆解、指纹提取）放开备选模型——这些任务对输出风格不敏感，风险最低；
2. 再放开 **规划类任务**（节拍规划、走向候选）；
3. 最后放开 **正文生成**，且必须伴随 A/B 度量（见 `06-开发方式与评估体系.md`）。

## 7. Token / 成本 / 延迟预算

| 场景 | 相对现状的开销 | 控制手段 |
|------|----------------|----------|
| G2 多候选（N=3~5） | 生成成本 ×N，评分另计（快速档启发式≈0） | 候选用低精细档评分先淘汰，仅幸存者进 LLM 精细评审；用户可配 N |
| G3 整本拆解（100 万字） | 一次性长任务，约为全文 token 的 1.2~1.5 倍 | 异步 + 分章增量 + 结果永久缓存（技法卡可跨作品复用） |
| G4 摄入 + 指纹 | 同上量级，一次性 | 同上；指纹构建支持"抽样章节"模式（如取 20% 代表章节） |
| G4 单次续写装配 | 上下文增大 30-60%（指纹 + 契约 + 检索片段） | 装配优先级裁剪（见 3.1）；最近原文窗口限定 2-3 章 |

计费侧无需新机制：全部走现有 `EconomyService` token 计量；建议为长任务（拆解/摄入）增加**预估报价 + 用户确认**步骤，避免误触大额消耗。

## 8. 与当前能力的依赖关系

```
Lorebook/图谱 ──────────────────────► G4 强依赖（实体/伏笔/时间线）；G3 弱依赖
风格画像/角色声音 ─────────────────► G3/G4 强依赖（指纹与技法的载体）
反 slop 质量门禁 ──────────────────► G2 强依赖（评委）；G4 强依赖（保真度监控）
模型策略 ──────────────────────────► G2-B / 全局 M1 档的扩展起点
批量异步任务体系 ──────────────────► G1/G3/G4 的前置件
```

结论：v3 不引入与当前实现冲突的架构；图谱完整化与通用批量异步任务体系仍是 G4/G3 的 P0 前置项。
