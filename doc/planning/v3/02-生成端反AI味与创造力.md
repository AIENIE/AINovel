## 1. 背景与痛点

### 1.1 现状：检测端已成熟，生成端仍是"裸奔"

现有反 AI 味体系是**事后防御型**：

```
生成（现有 prompt，仅含少量"避免AI套话"的稳定规则）
  → LocalSlopHeuristics 打分
  → SlopDiagnosticService AI 精细诊断（risk ≥ 55 时）
  → AiConservativeRevisionService 保守修订（仅局部、不改剧情）
  → 落库遥测
```

这套体系的三个天花板：

1. **保守修订只能"降害"不能"增益"**：它能删掉"嘴角微微上扬"，但不能把一段平庸的冲突写得让人想读下去。修订约束（不改剧情、不改角色决策）决定了它无法触及情节层的 AI 味。
2. **AI 味的大头不在词句层**：内部五维研究汇总指出，consistency、breath_focus_pacing、human_trace 的问题（事件传送带、缓冲感缺失、无代价爽点、情绪激活失败）在生成完成时已经定型，末端修订救不回来。
3. **根因是模型的模式坍缩（mode collapse）**：研究表明 RLHF 对齐使模型收敛到少数"安全同质"的输出吸引态，创意写作中的不确定性显著低于职业作家（见 05 号文档 §1.5）。只靠"不要写 X"的负面指令无法逆转分布坍缩，必须在**采样、规划、候选结构**上做文章。

### 1.2 AI 味的三层来源（本文档的核心认知框架）

| 层级 | 表现 | 现有系统覆盖 | v3 治理手段 |
|------|------|--------------|-------------|
| L1 表达层 | 套话、器官轮班、意象套件、"不是…而是…" | ✅ 检测+修订成熟 | 生成前注入负面约束（把检测词典搬到生成侧） |
| L2 情节层 | 事件传送带、无代价胜利、缓冲感缺失、钩子衰减、情绪激活失败 | ⚠️ 仅能检出，无法修复 | 节拍规划 + 情绪曲线目标 + 张力自检 |
| L3 规划层 | 题材同质、开局撞车、角色功能化、所有故事长一个样 | ❌ 完全未覆盖 | 规划级发散（多候选走向）、人设多样化、约束激发 |

**结论：G2 的主战场在 L2/L3，L1 只是顺手收编。**

### 1.3 业界与研究参考（详见 05 号文档）

| 来源 | 结论 | 对本方案的输入 |
|------|------|----------------|
| DOC / Re3 / Agents' Room / CogWriter | 先规划后成文 + 评审循环，连贯性 +22.5%、趣味性 +20.7%（DOC vs Re3） | 方案 A 的"节拍规划前置" |
| Verbalized Sampling（2025） | 让模型显式列出多个候选及其概率，多样性提升 2-3 倍，免训练 | 方案 A 的候选生成提示词范式 |
| min-p 采样（2024） | 创意任务显著优于 top-p；推荐 temp 0.8-1.2 + min-p 0.05 | 方案 A/B 的采样参数（需 AI 网关透传采样参数） |
| CritiCS（EMNLP 2024） | 多评论者协作精化计划与草稿，显著提升创意与读者参与度 | 方案 B 的多角色评审 |
| 人设多样性研究（2025） | "普通多样人设"优于名人人设，能推动模型进入知识空间不同区域 | 方案 B 的创意人格库设计原则 |
| 网文工艺研究 | 头部作品平均 1.8 章一个情绪高潮（低排名 4.7 章）；三层钩子；W/M/N 型情绪曲线 | 方案 A 的节拍规划目标函数 |
| McKee / Swain | 欲望-冲突-代价三要素；场景(目标→冲突→灾难)-序列(反应→困境→决定)节奏环 | 节拍规划的结构模板 |

## 2. 设计目标

| 编号 | 目标 | 度量（详见 06 号文档） |
|------|------|------|
| G2-1 | 生成文本的 slop 风险显著下降 | `slop_quality_runs` 风险分中位数 -20%；触发保守修订的比例 -30% |
| G2-2 | 读者主观吸引力提升 | 盲测中"更想读下去"得票率 ≥ 65%（vs 现有链路产物） |
| G2-3 | 产出多样性提升 | 同一大纲生成 10 次的候选间 distinct-3 / 语义距离显著高于基线 |
| G2-4 | 情节层质量可控 | 节拍计划中"冲突-代价-变化"三要素完备率 ≥ 90%（LLM 审计） |
| G2-5 | 不牺牲一致性 | 连续性/设定冲突类 issue 数量不高于基线（SlopQualityGate 监控） |

## 3. 方案对比

| 维度 | 方案 A：节拍规划 + 负面约束 + 多候选竞技场 | 方案 B：多模型创意流水线 + 创意人格 | 方案 C：偏好优化 / 风格微调轨道 |
|------|------|------|------|
| 核心策略 | 在锁定模型上重构生成管线：先发散规划、带约束起草、多候选评分选优 | 规划/起草/润色/评审由不同模型+人格分工，发散与收敛用不同引擎 | 用平台沉淀的偏好数据（候选选择、slop 标注）做 DPO/GRPO 或风格 LoRA，从权重层面矫正模式坍缩 |
| 模型要求 | M0（现状即可，需网关透传采样参数） | M1（放开 AiModelPolicy，复用当前模型路由能力） | M2（训练基础设施或外部微调 API） |
| 复杂度 | 中 | 中高 | 高 |
| 单章成本 | ×2 ~ ×4（规划 + N 候选） | ×2 ~ ×3（多阶段但可用廉价模型分担） | 训练一次性投入；推理成本≈现状 |
| 见效速度 | 快（纯工程） | 中（依赖模型解锁与路由建设） | 慢（数据积累 → 训练 → 评估） |
| 主要风险 | 提示词内卷：约束清单本身固化成新模板 | 多模型风格缝合感；成本控制复杂 | 数据质量决定上限；过拟合"评委口味" |
| 复用模块 | PromptAssemblyService、SlopQualityGate（评委）、LocalSlopHeuristics（快筛）、模式库 | 同左 + 当前模型注册/路由 | candidate_runs 数据、slop 标注、外部训练设施 |
| 关系 | 地基，B/C 都建立其上 | A 的增强 | A 的数据出口，长期最优解 |

**三个方案不是互斥关系，而是 A → B → C 的演进链：A 先把管线搭对；B 换更强的发动机；C 把平台数据变成护城河。**

## 4. 方案 A 详设：节拍规划 + 负面约束 + 多候选竞技场（推荐首发）

### 4.1 新生成管线

```
现状：场景摘要 ──────────────────────────► 正文（一步到位）

方案A：
场景摘要
  │
  ▼ ①节拍规划（Beat Planning）——攻击 L2/L3
  │   assembleBeatPlan()：产出 2-3 个差异化节拍方案（Verbalized Sampling 范式：
  │   要求模型输出"彼此策略不同"的候选并自述差异点）
  │   每个节拍方案必须显式回答：
  │   - 本场景主角想要什么（desire）/ 阻力是什么（conflict）/ 代价是什么（stakes）
  │   - 场景型还是序列型（Swain：目标→冲突→灾难 / 反应→困境→决定）
  │   - 情绪曲线位置（承接上文的压制/爆发周期，标记本场景是蓄力还是释放）
  │   - 结尾钩子类型（事件/人物/情绪三选一，含强度）
  │   自动选优或用户挑选（快速通道：LLM 按"三要素完备+与前文差异化"自动选）
  │
  ▼ ②带约束起草（Constrained Drafting）——收编 L1
  │   assembleWithCreativeConstraints()：
  │   - 注入选定节拍方案（硬约束）
  │   - 负面模式清单：从模式库采样轮换 15-25 条（防提示词固化，见 4.3）
  │   - 人味要素清单：正向要求（见 4.4）
  │   - 采样参数：temp 0.9-1.1 + min-p 0.05（需 AI 网关 gRPC 透传）
  │   同一节拍生成 N=2~3 个候选（差异化指令：不同的切入视角/开场时点）
  │
  ▼ ③竞技场选优（CandidateArena）
  │   快筛：LocalSlopHeuristics 全量打分（本地零成本），淘汰明显 slop
  │   精评：幸存候选走复合评分 = slop 风险(反向) + 节拍达成度 + 钩子有效性
  │        （LLM 评审，提示词强制引用原文证据，复用 slop 诊断的证据规范）
  │   选优产物照常进 SlopQualityGate 末端门禁（防线不撤）
  │
  ▼ 正文 + 候选与评分全量落库（candidate_runs/candidate_items，为方案 C 攒数据）
```

### 4.2 与现有代码的集成

| 触点 | 改动 |
|------|------|
| `PromptAssemblyService` | 新增 `assembleBeatPlan` / `assembleWithCreativeConstraints`（见 00 号文档 §3.1）；现有 `assembleSceneDraft` 保留为"快速模式" |
| `ManuscriptService.generateForScene` | 增加生成模式参数：`fast`（现状链路）/ `crafted`（方案 A 全链路）；用户按章选择 |
| `AiService` / AI 网关 gRPC | 需支持透传采样参数（temperature/top_p/min_p）——当前仅 chat 语义，属小改动但跨服务，需与 aiservice 协调 |
| `SlopQualityGate` | 抽出评分能力为可独立调用的 `scoreCandidate()`（内部复用现有启发式与诊断服务） |
| `LocalSlopHeuristics` | 无改动，直接作为快筛 |
| 模式库 | `V2__slop_patterns.sql` 建立并初始化 `slop_patterns` 表，供负面清单采样 |

### 4.3 负面约束的"采样轮换"设计（防止提示词自身成为新模板）

- 每次起草从模式字典按权重抽 15-25 条（高权重高概率，但非全量），且**按类别配额**（词句/动作链/意象/结尾套话/议论腔各若干）；
- 抽样种子与 sceneId 绑定，同场景重掷时更换种子，确保重掷真的"换个写法"；
- 定期用 `slop_quality_runs` 回归验证：若某类 issue 反弹，提升该类配额。

### 4.4 人味要素正向清单（从检测研究反推的生成目标）

来自 `03-algorithm-ideas.md` 的 human_trace 维度与网文工艺研究，起草提示词要求正文至少体现其中若干项：

1. **具体经验质感**：用只有"在场者"才知道的细节替代通用描写（不是"疼痛袭来"，而是这种疼具体像什么、影响了哪个动作）；
2. **代价可见**：本场景的收获必须有对应付出（信息差、关系损耗、资源消耗），禁止无代价爽点；
3. **关系向量变化**：至少一组人物关系在本场景结束时与开始时不同（哪怕微小）；
4. **呼吸感**：高强度事件后必须有消化性瞬间（缓冲），禁止事件传送带；
5. **主体聚焦**：视角人物有连续的注意力逻辑，不做全知式扫描。

这份清单同样采样轮换（每次强调 2-3 项），避免每章都均匀塞满五要素造成新的公式化。

### 4.5 数据模型（草案）

```sql
CREATE TABLE creativity_profiles (
    id              CHAR(36) NOT NULL,
    user_id         CHAR(36) NOT NULL,
    story_id        CHAR(36) NULL COMMENT 'NULL 为全局默认',
    divergence      TINYINT  NOT NULL DEFAULT 5  COMMENT '发散强度 1-10：影响节拍候选差异度与采样温度',
    candidate_count TINYINT  NOT NULL DEFAULT 3  COMMENT '每场景候选数 N（成本开关）',
    negative_strictness TINYINT NOT NULL DEFAULT 5 COMMENT '负面清单严格度：影响抽样条数',
    human_trace_prefs_json TEXT NULL COMMENT '人味要素偏好权重',
    beat_review_mode ENUM('auto','manual') NOT NULL DEFAULT 'auto',
    PRIMARY KEY (id)
);

CREATE TABLE candidate_runs (
    id          CHAR(36) NOT NULL,
    scene_ref   VARCHAR(128) NOT NULL,
    stage       ENUM('beat_plan','draft') NOT NULL,
    winner_item CHAR(36) NULL,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);

CREATE TABLE candidate_items (
    id          CHAR(36) NOT NULL,
    run_id      CHAR(36) NOT NULL,
    content     LONGTEXT NOT NULL,
    heuristic_score INT NULL,
    composite_score_json TEXT NULL COMMENT 'slop反向分/节拍达成/钩子有效性/评审引证',
    chosen_by   ENUM('auto','user') NULL,
    PRIMARY KEY (id)
);
```

`candidate_items` 是方案 C 的训练数据金矿：每一行"被选中 vs 被淘汰"天然构成偏好对。

### 4.6 成本与延迟

- `crafted` 模式单章成本约为现状 ×2.5~×4（1 次规划 + 2-3 候选 + 1 次精评；快筛免费）。
- 延迟：候选并行生成，端到端约为现状 ×1.5~×2；批量模式下走 AsyncJobService 无感。
- 成本开关全部暴露给用户（candidate_count、beat_review_mode），并计入现有 EconomyService 计费。

## 5. 方案 B 详设：多模型创意流水线 + 创意人格

### 5.1 形态

在方案 A 管线上做两个替换：

1. **阶段换模型**（采用能力标签与任务路由规则）：
   - 节拍规划/走向发散 → 高发散创意模型；
   - 正文起草 → 中文文学质量最优的模型；
   - 竞技场评审 → 严谨分析型模型（评审与起草不同源，避免"自评自"偏置——LLM 评审偏好自家文本是已知缺陷，见 05 号文档 §3.1）；
   - 保守修订 → 廉价快速模型。
2. **创意人格库**（`CreativePersonaInjector`，见 00 号文档 §3.2）：起草阶段为候选分配不同叙事者气质（如"克制的白描者 / 热衷感官细节者 / 对话驱动者"）。设计原则采纳人设多样性研究结论：**用普通而互异的人格，不用名人人格**；人格与 StyleProfile 正交（画像管"这本书的风格"，人格管"这个候选的切入气质"），选优后人格痕迹被画像约束收敛。

### 5.2 前置条件与风险

- 前置：`AiModelPolicy` 解锁（按 00 号文档 §6.2 的三步走：先评审类→再规划类→最后正文）；AI 网关支持多模型计费。
- 风险：多模型正文的"缝合感"——缓解：正文起草始终单模型，多模型只用于规划与评审；跨章一致性由 StyleProfile + SlopDriftService 监控。

## 6. 方案 C 详设：偏好优化 / 微调轨道

### 6.1 数据资产（平台独有优势）

| 数据 | 来源 | 用途 |
|------|------|------|
| 偏好对（chosen vs rejected） | `candidate_items`（方案 A 运行即积累） | DPO 训练对 |
| slop 细粒度标注 | `slop_quality_runs` / `slop_quality_issues`（含 span 级 quote 与维度） | 负例挖掘、reward 信号 |
| 用户真实修改 diff | 当前稿件版本与编辑历史 | 最高价值信号：用户实际不满意什么 |

### 6.2 两条训练路线

1. **反 slop 偏好优化（DPO/GRPO）**：目标不是学会某种风格，而是把模式坍缩"掰开"——奖励在保持连贯下的表达多样性与三要素完备。GRPO 已被证明可用于长篇作者风格保持（见 05 号文档 §2.8）。
2. **平台文风 LoRA 族**：按题材/平台调性（快节奏爽文 / 细腻女频 / 文学向）训练少量 LoRA，作为"底味"选项；与 G4 的单作品 LoRA（04 号文档方案 C）共享训练管线。

### 6.3 风险与门槛

- 依赖 M2 基础设施（自建训练或外部微调 API）与模型许可（deepseek 系列权重可获得性 / 或改用可微调的开源底座——这本身是重大架构决策）；
- 评估必须用**人类盲测**兜底：DPO 很容易过拟合"评委口味"，出现指标涨、体感差；
- 建议作为 6-12 个月期的储备轨道：方案 A 上线即开始攒数据，数据到位（≥1 万偏好对）再立项训练。

## 7. 推荐路径

1. **首发方案 A**：不依赖任何外部决策（模型锁定下可行），直接攻击 L2/L3，且为 C 攒数据。其中 AI 网关透传采样参数是唯一的跨服务小改造，建议尽早排期。
2. 方案 B 在 `AiModelPolicy` 解锁后作为 A 的增强包迭代（评审异源化 → 规划高发散化 → 正文换模型）。
3. 方案 C 长期储备，触发条件：偏好对数据 ≥1 万 且 A/B 的盲测提升进入平台期。
4. 与其他目标的联动：G1 向导的每一步候选、G3 的技法注入、G4 的续写起草全部复用本方案的 CandidateArena 与负面约束基建。
