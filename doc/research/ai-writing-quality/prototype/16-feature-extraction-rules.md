# 16. Feature Extraction Rules：正则 / 语义 / LLM 判断特征提取规则

本文件定义 MVP 检测器如何从中文小说文本中提取 slop / 模板化风险特征。核心原则：**能用规则稳定抓的用规则；必须理解语境的交给 LLM；所有特征都要能回溯原文证据。**

---

## 1. 特征提取三层

| 层级 | 方法 | 适合特征 | 优点 | 风险 |
|---|---|---|---|---|
| L1 | 正则 / 关键词 | 固定句式、动作链、元泄漏、Markdown残留 | 稳定、便宜、可重复 | 容易误判语境 |
| L2 | 统计 / 结构规则 | 密度、聚集、共现、对话尾巴比例 | 可量化 | 需要窗口和阈值调参 |
| L3 | LLM 语义判断 | 语域、角色贴合、事件传送带、Human Trace | 能理解语境 | 成本高、需约束输出 |

---

## 2. 统一特征格式

无论来自规则还是 LLM，最后都转成统一结构：

```json
{
  "feature_id": "F-0001",
  "source": "regex|keyword|statistic|llm",
  "module": "surface_template",
  "category": "phrase_pattern",
  "pattern_id": "A001",
  "quote": "不是愤怒，而是一种更深的失望",
  "char_start": 120,
  "char_end": 137,
  "window_id": "w1",
  "base_weight": 2,
  "raw_confidence": 0.85,
  "explanation": "500字内第3次出现排除式升级句式",
  "repair_hint": "改成角色具体行动或判断"
}
```

---

## 3. L1：正则 / 关键词规则

### 3.1 固定句式

#### A001：不是……而是……

建议正则：

```regex
不是[^。！？；\n]{0,30}而是
```

捕捉：

- 不是愤怒，而是一种更深的失望；
- 不是害怕，而是某种近乎麻木的平静。

降权条件：

- 非小说正文；
- 角色正在辩论、推理、讲课；
- 单次出现。

升权条件：

- 500 字内 3 次以上；
- 与“语气里带着”“声音里透着”等共现；
- 用于抽象总结情绪。

#### A002：没有……也没有……而是……

```regex
没有[^。！？；\n]{0,25}(?:也没有|更没有)[^。！？；\n]{0,40}而是
```

#### A003：不是那种……而是一种更……

```regex
不是那种[^。！？；\n]{0,40}而是(?:一种)?更[^。！？；\n]{0,40}
```

---

### 3.2 语气描写模板

关键词 / 短语：

```text
语气里带着
声音里透着
嗓音有些
语调变得
用一种……的口吻
像在说今天天气
仿佛在阐述一个既定事实
```

建议规则：

```regex
(语气里带着|声音里透着|嗓音有些|语调变得|口吻|既定的事实|今天天气怎么样)
```

额外统计：

- 台词句数；
- 台词后跟语气/神态/心理解释的比例；
- 同一场景内是否每句台词都拖尾巴。

---

### 3.3 身体动作链 / 器官轮班

关键词组：

```yaml
body_actions:
  - 指尖轻轻敲
  - 指尖无意识
  - 指节泛白
  - 指尖泛白
  - 死死咬住下唇
  - 咬住下唇
  - 喉咙发紧
  - 喉结滚动
  - 耳尖泛红
  - 耳根泛红
  - 摩挲
  - 推了推眼镜
  - 拿起咖啡
  - 拿起烟
  - 嘴角勾起
  - 眼中闪过
  - 瞳孔收缩
  - 瞳孔放大
```

规则重点：

- 单个动作不判高风险；
- 统计动作密度；
- 统计动作是否推动剧情；
- 多个角色是否共用同一套动作；
- 与角色设定冲突时升级，例如无眼镜角色推眼镜。

---

### 3.4 意象 / 比喻套件

关键词组：

```yaml
imagery_cliches:
  - 石子投入湖面
  - 涟漪
  - 影子拉得很长
  - 阳光透过窗户
  - 光斑
  - 轻得像羽毛
  - 小兽般
  - 离水的鱼
  - 煮熟的虾
  - 手术刀般
  - 浮木
  - 超新星爆发
  - 世界安静下来
```

提取规则：

- 匹配 phrase；
- 统计章节开头/结尾出现位置；
- 与抽象情绪总结共现时升权；
- 若意象与角色经验系统无关，由 LLM 二次判断。

---

### 3.5 升华 / 收尾套话

关键词：

```yaml
ending_cliches:
  - 这就够了
  - 夜还很长
  - 不急，慢慢来
  - 才刚刚开始
  - 游戏才刚刚开始
  - 此刻，无人知晓
  - 注定无法平静
  - 无人能够预料
  - 传奇永不落幕
```

提取要求：

- 如果位于章节最后 10% 文本，权重增加；
- 如果多个章节都以同类抽象留白结尾，权重增加；
- 如果前文有具体未解决钩子，则降权。

---

### 3.6 元提示 / 生成残留

高置信关键词：

```yaml
meta_leaks:
  - 以下是修改后的版本
  - 好的，这就为您
  - 根据你的要求
  - 根据你的大纲
  - 用户说过
  - 作为一个AI
  - 我无法
  - ###
  - ---
  - **
```

注意：Markdown 标记在聊天体、论坛体、游戏系统面板中可能合理，需要结合文本类型。

证据等级：

- 明确“用户/指令/修改版/AI” → E4；
- Markdown 标记突兀出现在正文 → E3/E4；
- 论坛体合理使用 Markdown → 降权。

---

## 4. L2：统计 / 结构规则

### 4.1 窗口密度

默认窗口：

```yaml
window_sizes:
  short: 500
  medium: 1000
  long: 3000
```

指标：

```text
pattern_density = pattern_count / char_count * 1000
category_density = category_count / char_count * 1000
```

初始阈值：

| 指标 | medium | high |
|---|---:|---:|
| 表层 pattern / 1000字 | 6 | 12 |
| 动作链 / 1000字 | 4 | 8 |
| 意象套件 / 1000字 | 2 | 5 |
| 升华套话 / 章末 | 1 | 2 |
| 语气描写 / 台词数 | 30% | 60% |

### 4.2 共现 / 共振

共振升权条件：

```text
surface_resonance = phrase_pattern + tone_description + body_action + imagery_cliche
```

示例：

- 500字内 3类以上表层 pattern；
- 同一段同时有抽象情绪总结 + 器官轮班 + 意象套件；
- 章节结尾重复使用抽象升华。

### 4.3 对话尾巴比例

定义：

```text
dialogue_tail = 台词结束后紧跟的神态/动作/心理/环境解释
```

计算：

```text
dialogue_tail_ratio = 带尾巴台词数 / 总台词数
```

初始阈值：

- 0.30 以下：正常；
- 0.30-0.60：中风险；
- 0.60 以上：高风险，尤其是每句台词都跟解释。

### 4.4 事件传送带粗提取

规则层只能粗提取：

- 段落中高频出现“突然、就在这时、下一秒、还没等、紧接着、与此同时”；
- 场景切换密集；
- 冲突动词密集；
- 缺少表示后果/休整/复盘的词。

最终仍需 LLM 判断。

## 5. L3：LLM 语义判断规则

LLM 必须使用结构化输出，且每条判断引用原文。

### 5.1 语域适配判断

输入：

- 文本；
- 角色档案；
- 题材/平台；
- POV；
- 世界观。

判断：

1. 台词是否像该角色会说；
2. 内心独白是否符合身份、教育、年龄、时代；
3. 旁白是否越界成分析腔；
4. 同人文是否贴原作角色；
5. 是否有平台/商务/议论文词侵入。

输出字段：

```json
{
  "category": "voice_fit",
  "quote": "...",
  "issue": "旁白突然变成议论文分析腔",
  "severity": "medium",
  "alternative_explanations": ["角色正在理性分析"],
  "repair_hint": "改回角色视角下的感知和判断"
}
```

### 5.2 一致性与设定吸收判断

判断：

- 是否违反角色档案；
- 是否未授权新增道具/能力/设定；
- 是否前后事实矛盾；
- 是否只抓人名关键词，没有让设定影响选择和代价。

LLM 必须指出“违反了哪条设定”。

### 5.3 Breath / Focus / Pacing 判断

判断：

1. 事件是否机械续接；
2. 每个事件后是否有后果消化；
3. 缓冲段是否承担剧情功能；
4. 主角是否只有数值成长；
5. 读者注意力是否被同权描述打散。

输出必须包含事件链。

### 5.4 Human Trace 判断

判断：

- 是否有具体经验；
- 是否有作者取舍；
- 是否能看出作者真正想写的关系/爽点/痛点；
- 角色选择是否可替换；
- 文风是否过度正确平滑。

注意：Human Trace 低不是“作者不是人”，而是文本缺少不可替代性。

---

## 6. 特征到证据等级映射

| 特征来源 | 默认等级 | 升级条件 |
|---|---|---|
| 单个固定句式 | E1 | 500字内高密度、多类共现 → E2 |
| 单个动作链 | E1 | 与角色设定冲突 → E3 |
| 意象套件 | E1 | 多个意象套件聚集、章末反复 → E2 |
| 语气描写模板 | E1 | 台词尾巴比例高 → E2 |
| 事件传送带 | E2 | 多章节持续且导致角色无成长 → E3 |
| 设定吸收失败 | E3 | 多处严重违反核心设定 → E3 high |
| 近句事实矛盾 | E3 | 影响剧情逻辑 → E3 high |
| 元提示泄漏 | E4 | 直接强证据 |

---

## 7. 误判控制

### 7.1 必须降权的情况

- 角色本来就是分析型、教师型、律师型；
- 叙事体裁是论坛体、系统面板、报告体；
- 题材传统本来依赖某些俗套表达；
- 高潮段快节奏不等于事件传送带；
- 短篇为了压缩可以少写缓冲；
- 搞笑/反讽文本故意使用套话。

### 7.2 必须输出替代解释

至少从以下选项中选：

```text
genre_cliche
studio_formula
human_low_skill
author_style
platform_pressure
intentional_pastiche
missing_context
```

---

## 8. 实现建议

### 8.1 规则优先级

先实现：

1. 元泄漏；
2. 固定句式；
3. 动作链；
4. 意象套件；
5. 升华收尾；
6. 对话尾巴比例；
7. 密度统计。

再实现：

1. 语域适配 LLM；
2. 一致性 LLM；
3. 呼吸感 LLM；
4. Human Trace LLM。

### 8.2 LLM 调用策略

- 文本少于 3000 字：一次性诊断；
- 文本 3000-20000 字：按窗口诊断后汇总；
- 长篇：只抽样窗口，做曲线比较；
- 规则高风险窗口优先送 LLM，节省成本。

### 8.3 输出约束

LLM 输出必须是 JSON 或严格 Markdown 表格；每条判断必须有：

- quote；
- issue_type；
- severity；
- alternative_explanations；
- repair_hint。
