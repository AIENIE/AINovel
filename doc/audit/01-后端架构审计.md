# 后端架构审计（A 系列）

范围：`backend/src/main/java/com/ainovel/app`（285 个文件，约 24,280 行，20 个顶层包）。

## A-01 V2 控制器双实现/内存回退（P1，临时绕过逻辑）

**位置**：
- `backend/src/main/java/com/ainovel/app/v2/V2VersionController.java`（829 行）
- `backend/src/main/java/com/ainovel/app/v2/V2ExportController.java`（766 行）
- `backend/src/main/java/com/ainovel/app/v2/V2WorkspaceController.java`（641 行）
- `backend/src/main/java/com/ainovel/app/v2/V2ContextController.java`（633 行）
- `backend/src/main/java/com/ainovel/app/v2/V2ModelController.java`（442 行）
- `backend/src/main/java/com/ainovel/app/v2/V2AnalysisController.java`（253 行）

**问题**：六个 V2 控制器均内嵌一套完整的 ConcurrentHashMap"内存版"实现，仅当注入的持久化服务为 `null` 时启用。以 V2VersionController 为例：

- 内存状态字段：`V2VersionController.java:32-36`（`branchByManuscript`、`versionByManuscript`、`diffCache`、`autoSaveByUser`、`initLocks` 共 5 个 ConcurrentMap）；
- 双构造器：`V2VersionController.java:38-40` 为非 `@Autowired` 便捷构造器（service 传 null），`:42-49` 为 Spring 实际使用的 `@Autowired` 构造器；
- 空值分支：13 处 `if (versionService != null)`（`:58, 73, 114, 133, 193, 252, 267, 300, 354, 383, 485, 503, 515`）。

```java
// V2VersionController.java:58-63
if (versionService != null) {
    return versionService.listVersions(manuscript, user);
}
ensureMainBranchAndInitialVersion(manuscript, user);
return sortedVersions(manuscriptId);
```

V2ExportController 同型（2 个内存 Map `:42-43`，10 处空值分支），V2WorkspaceController 同型（4 个内存 Map `:30-33`，10 处空值分支）。

**证实的关键事实**：
1. 全部 6 个 `V2*PersistenceService` 均为无条件注册的 `@Service`（如 `V2VersionPersistenceService.java:24`、`V2ExportPersistenceService.java:17`），无任何 `@ConditionalOn*` —— **生产环境回退分支永远不可达，是死代码**；
2. 便捷构造器仅被测试使用（`backend/src/test/java/com/ainovel/app/v2/V2VersionControllerTests.java:37`、`V2ExportControllerTests.java:36`、`V2WorkspaceControllerTests.java:39`）—— 即数千行内存实现本质上是**嵌在生产代码里的测试脚手架**。

**影响**：
- 六个控制器合计约 3,500 行，估计 40%-60% 属于死的回退路径，直接推高理解与修改成本；
- 若未来有人在非 Spring 场景实例化控制器，内存路径会**静默丢数据**（重启即失、多实例不一致）；
- 每个接口两条执行路径，测试验证的（内存路径）与生产运行的（持久化路径）**不是同一套代码**，测试有效性打折。

**违反原则**：KISS（双路径复杂性）、YAGNI（生产不需要的内存实现）、单一职责（控制器同时承担 HTTP 编排 + 存储实现）。

**整改建议**：仓库内已有正确范例——`quality/SlopQualityRecorder` 接口 + `JpaSlopQualityRecorder`（生产）+ `InMemorySlopQualityRecorder`（测试，位于生产包但为独立小类）。建议：
1. 删除六个控制器中的内存字段、便捷构造器与全部 `!= null` 分支，持久化服务改为 `final` 必注入；
2. 控制器测试改为 mock 持久化服务，或将内存实现抽为测试目录下的 Fake 实现类；
3. 顺带解决 A-05（回退路径删除后，返回值可统一 DTO 化）。

## A-02 分层违规：控制器直连 Repository 与 currentUser 复制（P1）

**位置与证据**：
- `story/StoryController.java:38` 注入 `StoryRepository`；`:42-44` 私有 `currentUser()`；
- `user/UserController.java:40` 注入 `UserRepository`；`:53-55` 同型 `currentUser()`；
- `settings/SettingsController.java:23` 注入 `UserRepository`；`:26` 同型辅助；
- `world/WorldController.java:26`、`v2/V2VersionController.java:28`（`ManuscriptRepository`）等同类注入；
- `v2/V2AccessGuard.java:28-33` 又实现了一份带中文错误消息的 `currentUser()`。

```java
// StoryController.java:42-44 与 UserController.java:53-55 逐字符相同
private User currentUser(UserDetails details) {
    return userRepository.findByUsername(details.getUsername()).orElseThrow();
}
```

**影响**：领域规则（"当前用户如何解析"、"资源归属如何校验"）散落在控制器层；`findByUsername` 每请求查库且实现多处复制，改一处漏一处（如错误消息，一处 `orElseThrow()` 裸抛、一处抛"用户不存在"）。

**违反原则**：DRY、分层架构（Controller → Service → Repository）。

**整改建议**：抽取单一的 `CurrentUserResolver`（或 `HandlerMethodArgumentResolver` 直接把 `User` 注入控制器方法参数），统一错误语义；控制器持有的 Repository 一律下沉到对应 Service。

| 类 | 行数 | 注入依赖 | 混杂职责 |

**影响**：ManuscriptService 是生成链路的总枢纽，v3 G2 要在 PromptAssemblyService 新增 5 个装配入口（v3-proposals/00 §3.1），全部要经此类接线——不先拆分，v3 每个目标都会继续往里加依赖。

**整改建议**：按职责切分，例如 ManuscriptService → `SceneGenerationService`（生成编排：AI + 提示词 + 风格 + 质量门禁）、`ManuscriptContentService`（章节内容与角色变更日志 CRUD）；StoryService 拆出 `CharacterCardService`。拆分应发生在 v3 G2 扩展 PromptAssemblyService 之前（见 04 路线图阶段二）。

## A-04 真 N+1 查询（P1，性能）

**位置与证据**：

1. `user/UserController.java:235-244`，`estimateTotalWords()`——用户 profile/summary 接口每次调用触发：

```java
for (Story story : storyRepository.findByUser(user)) {            // 1 次
    for (Outline outline : outlineRepository.findByStory(story)) { // N 次
        for (Manuscript m : manuscriptRepository.findByOutline(outline)) { // N×M 次
            total += estimateWordsFromSections(m.getSectionsJson());
```

10 故事 × 10 大纲 × 5 稿件 ≈ 100+ 条查询/请求，且发生在高频的个人信息接口上。

2. 关联的性能模式（非 N+1，但同根）：`material/MaterialService.java:223-246` 素材引用扫描——`findByStoryUser` 已做 `join fetch`（`ManuscriptRepository.java:20`，复核确认**不是** N+1），但每次引用查询会把该用户**全部稿件的 sectionsJson 全文**载入内存逐段扫描，数据量增长后延迟与内存线性恶化。

以下初判嫌疑经复核**不属于**真 N+1（避免误伤）：`MaterialService.sceneInfoById`（outline 已 fetch join）、`StyleService.java:77`（单查询后内存迭代）、`OutlineService.java:395-407`（`findAllWithStoryUser` 单查询 + JSON 内存遍历，但注意该方法做全表扫描，数据量增长后需分页）。

**整改建议**：`estimateTotalWords` 改为一条聚合 JPQL/原生 SQL（join 三表取 sectionsJson 或预先冗余字数字段）；素材引用扫描中期改为写路径维护的引用索引/物化统计，避免读路径全文扫描。

## A-05 API 响应约定分裂（P1）

**证据**：
- v1 控制器返回类型化 DTO：`StoryController` 返回 `List<StoryDto>`、`ManuscriptController` 返回 `ManuscriptDto`、`UserController` 返回 `ResponseEntity<UserProfileResponse>`；
- v2 控制器全部返回裸 Map：`V2VersionController.java:54`（`List<Map<String, Object>>`）、`:68`（`Map<String, Object>`）、V2ExportController `:61, 126, 151, 172` 同型；
- 同时混用三种包裹风格：裸返回 / `ResponseEntity<T>` / `ResponseEntity<Void>`。

**影响**：v2 接口无编译期契约，前端 `mock-api.ts` 被迫写 200+ 行手工归一化函数（`toOutline`、`toPlotPlanning` 等，见 F 系列）；OpenAPI 文档（springdoc 已接入）对 Map 返回值无法生成有效 schema。

**整改建议**：v2 控制器逐个补 record DTO（与 A-01 的回退删除同批做工作量最小——Map 的存在正是为了让内存路径与持久化路径共用返回结构）。新端点一律禁用 `Map<String,Object>` 返回。

## A-06 全局异常处理不足（P1）

**位置**：`common/GlobalExceptionHandler.java`（37 行）。

**证据**：现有处理器仅 4 个（`:13-35`）：BadCredentials→401、UsernameNotFound→404、MethodArgumentNotValid→400、**RuntimeException→400（兜底）**。问题：
1. 兜底处理器把 NPE、IllegalState 等编程错误也映射为 400"客户端错误"，且**无任何日志**——线上 5xx 级故障会以 400 静默吞掉，无堆栈可查；
2. 缺失处理器：gRPC `StatusRuntimeException`（BillingGrpcClient/AiGatewayGrpcClient 上抛）、`DataIntegrityViolationException`（唯一键冲突）、`IllegalArgumentException/IllegalStateException`（EconomyService.java:132 等使用）；
3. 业务层大量用 `throw new RuntimeException("中文消息")` 表达业务错误（A-01/A-02 中多处引用），异常类型无法区分"业务拒绝"与"程序缺陷"。

**整改建议**：引入 `BusinessException`（或复用 ResponseStatusException）承载业务错误；兜底处理器改为 500 + `log.error` 记录堆栈；补 gRPC 与数据完整性异常映射。存量 `RuntimeException("消息")` 可用一次机械替换收口。

## A-07 ObjectMapper 分散实例化（P2）

**证据**：14 个类各自 `new ObjectMapper()`：MaterialService(:35)、PlotQualityController(:35)、UserController(:51)、WorldService(:32)、V2Json(:11)、OutlineService(:30)、V2ExportController(:40)、ManuscriptService(:67)、V2VersionController(:30)、StoryService(:37)、SlopQualityController(:38)、V2VersionPersistenceService(:31)、ExternalSecurityStartupValidator(:23)、QdrantMaterialVectorIndex(:21)。注入 Spring 管理实例的为 0 处（对照：`JpaSlopQualityRecorder` 构造器注入 ObjectMapper，是全库唯一正确用法）。

**影响**：各实例配置不一致的风险（时间格式、未知字段策略）；Spring Boot 的 `spring.jackson.*` 配置对这些实例全部失效。

**整改建议**：统一构造器注入 Spring 的 ObjectMapper；`V2Json` 静态工具改为注入式组件或删除。

## A-08 gRPC 客户端样板重复（P2）

**证据**：`integration/BillingGrpcClient.java`（428 行）与 `integration/AiGatewayGrpcClient.java`（342 行）在端点解析、通道复用判断、Metadata 鉴权头、stub 创建上逐段平行（BillingGrpcClient:291-331 的 `getOrCreateClient()` 与 AiGatewayGrpcClient:138-150 同构），重复约 50-60 行。`GrpcChannelFactory` 只抽了通道创建一层。

**整改建议**：抽 `AbstractGrpcClient<TStub>` 基类或组合式 `GrpcEndpointManager`（持有端点解析 + 通道生命周期 + 元数据装配），两个客户端只保留各自的业务方法。v3 若解锁多模型路由（M1 档）会新增第三个外部服务客户端，届时此抽象直接复用。

## A-09 双守卫体系（P2）

**证据**：`security/ResourceAccessGuard.java:38-50`（`assertOwner`，基于用户名比对 + admin 放行）与 `v2/V2AccessGuard.java:35-52`（`requireOwnedManuscript`，基于 ID 链比对）是两套并行的资源归属校验，错误消息、放行规则（V2AccessGuard 无 admin 放行）各不相同。

**整改建议**：合并为单一 Guard（保留 admin 放行语义并显式决定 v2 是否适用），v2 的 `requireOwned*` 系列作为其上的类型化便捷方法。

## A-10 @Transactional 出现在控制器（P1）

**证据**：`admin/ops/AdminOperationsController.java:111, 126, 138, 150, 166` 五个控制器方法标注 `@Transactional(readOnly = true)`。

**影响**：事务边界属于服务层职责；控制器上的事务会把 HTTP 序列化阶段也纳入事务窗口，且与全局约定（142 处 @Transactional 均在服务层）不一致。

**整改建议**：将查询逻辑下沉到 AdminOpsService（现有 ops 服务类可承接），控制器只做参数绑定与响应组装。

## A-11 JSON-in-TEXT 列与解析辅助重复（P2）

**证据**：Manuscript（sectionsJson、characterLogsJson）、Outline（contentJson）、Material（tagsJson、entitiesJson）、StyleProfile、World（modulesJson）、Story 六类实体以 TEXT 存 JSON；各服务重复实现 `readJson/readStringMap/writeJson` 私有辅助（ManuscriptService、OutlineService、WorldService、MaterialService、UserController:250/274 等），并与 A-07 的分散 ObjectMapper 叠加。

**影响**：JSON 结构变更无 schema 保障（ManuscriptService.java:220 已出现"Skip malformed historical data"的防御注释）；统计类需求被迫全量拉 TEXT 解析（与 A-04 相互放大）。

**整改建议**：短期抽公共 `JsonColumnConverter`（JPA `@Convert` 或统一工具类）消除辅助方法重复；中期对查询/统计高频的结构（如 sections 字数）增加物化列。JSON 列本身保留与否属产品权衡，不强制。

## 附：包结构总览（供整改定位）

20 个顶层包中，`v2`（55 文件）与 `quality`（约 40 文件）最大。`v2` 包并非 v1 的平行重写（无对应的 v1 VersionController 等），而是**纯新增功能集**——因此 A-01 的清理不存在"要与 v1 版本合并"的额外成本。`common` 包仅 3 个文件（GlobalExceptionHandler、ApiError 等），公共抽象层明显偏薄，A-02/A-07/A-11 的新公共组件可落位于此。
