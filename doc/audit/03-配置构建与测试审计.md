# 配置·构建·数据·测试审计（C / T 系列）

范围：`env.txt`、`application.yml`、`docker-compose.yml`、`build.sh`、`docker/`、`backend/sql/schema.sql`、`backend/pom.xml`、前后端测试。按用户要求，env.txt 的密钥暴露不作为发现；以下均为结构性问题。

## C-01 ddl-auto 漂移（P0）

**证据链**：
- `backend/src/main/resources/application.yml:11` 字面硬编码 `ddl-auto: update`（**未**参数化为 `${SPRING_JPA_HIBERNATE_DDL_AUTO:...}`）；
- `env.txt:67` 设 `SPRING_JPA_HIBERNATE_DDL_AUTO=none`，仅凭 Spring 松弛绑定的**环境变量优先级**覆盖 yml；
- 各运行方式的实际取值：

| 运行方式 | env.txt 是否生效 | 实际 ddl-auto |
|----------|------------------|---------------|
| `build.sh` Docker 部署 | 是（compose `--env-file` + 容器入口 `load-env-file.sh`，docker-compose.yml:10） | none ✔ |
| VSCode F5 调试 | 是（backend/.vscode 的 prepare-env.sh 加载） | none ✔ |
| **裸 `mvn spring-boot:run`** | **否** | **update ✘** |
| `mvn test` | 使用测试资源配置 | update（src/test/resources/application.yml:12）/ create-drop（application-test.yml:9） |

**风险**：application.yml 的数据源默认值指向远程库（`:6` 为 `base.seekerhut.com:23306`）。裸启动 = 以 `update` 模式连上真实远程库，Hibernate 会按实体定义**静默 ALTER/CREATE**——与 README"结构以 `backend/sql/schema.sql` 和显式变更为准"的声明直接矛盾。这是全部发现中唯一的数据风险项。

**整改建议**：`application.yml:11` 改为 `ddl-auto: ${SPRING_JPA_HIBERNATE_DDL_AUTO:none}`（默认收紧为 none）。一行修复，立即可做。

## C-02 env.txt 死键与双键回退（P2）

**死键**（全仓 grep 零引用，共 7 个）：`APP_TEST_DOMAIN`(env.txt:4)、`APP_PROD_DOMAIN`(:5)、`PIN_UPSTREAM_SERVICES_TO_LOCALHOST`(:6)、`USERSERVICE_HOST_IP`(:7)、`PAYSERVICE_HOST_IP`(:8)、`AISERVICE_HOST_IP`(:9)、`DEPS_AUTO_BOOTSTRAP`(:83)。另 `SPRING_JPA_PROPERTIES_HIBERNATE_DIALECT`(:68) 无对应的 yml 占位符（依赖 Spring 松弛绑定直连，属可用但隐晦）。

**双键回退**（application.yml 4 处，右侧 APP_* 键在 env.txt 与代码中均不存在，为不可达分支）：

```yaml
# application.yml:73-78
caller: ${EXTERNAL_AI_HMAC_CALLER:${APP_EXTERNAL_AI_HMAC_CALLER:}}
secret: ${EXTERNAL_AI_HMAC_SECRET:${APP_EXTERNAL_AI_HMAC_SECRET:}}
internal-token: ${EXTERNAL_USER_INTERNAL_GRPC_TOKEN:${APP_EXTERNAL_USER_INTERNAL_TOKEN:}}
service-jwt: ${EXTERNAL_PAY_SERVICE_JWT:${APP_EXTERNAL_PAY_SERVICE_JWT:}}
```

**整改建议**：删除 7 个死键（若为未来预留，注释标明用途与启用条件）；双键回退收敛为单键。

## C-03 无数据库迁移框架（P1）

**证据**：pom.xml 无 Flyway/Liquibase；`backend/sql/schema.sql`（831 行、49 张表）以日期注释分段手工维护（v1 基础 :1-167、v2 增量 2026-02-17 :168-540、经济账本 2026-02-20 :741-830）；线上依赖 `ddl-auto=none` + 人工执行 DDL。

**影响**：schema 变更无版本、无校验、无回滚脚本；与 C-01 组合后，"实体定义"与"schema.sql"哪个是真相取决于运行方式。**v3 数据模型草案新增 10+ 张表**（v3-proposals/00 §5：creation_workflows、candidate_runs、reference_works、author_fingerprints 等）——在此之前必须有迁移机制，否则手工 DDL 的错误面随表数线性放大。

**整改建议**：引入 Flyway：现有 schema.sql 整理为 `V1__baseline.sql`（生产库用 `flyway baseline` 对齐），之后所有变更走版本化迁移。列为 v3 立项的前置条件（见 04）。

## C-04 孤儿表（P2）

**证据**（精确核对：49 张表 vs 46 个 `@Entity` + 1 个 `@CollectionTable`）：
- `email_verification_codes`、`context_snapshots` 两张表在全部 Java 代码中**零引用**（无实体、无原生 SQL）；
- 其余 47 张表全部有对应实体/集合表映射（user_roles 由 `user/User.java:40` 的 `@CollectionTable` 覆盖）。

推测：邮箱验证已由 user-service（gRPC，proto 中有 EmailVerificationService）承接；context_snapshots 为 V2 上下文功能的预留表但实现未落库。

**整改建议**：确认后从 schema.sql 移除或注释标注"预留给 XX"，避免误导后续开发（尤其 v3 G4 要在上下文/快照方向扩表）。

## C-05 Redis 配置完整但零使用（P2）

**证据**：pom.xml:41 引入 `spring-boot-starter-data-redis`；application.yml:17-25 配置连接与 `spring.cache.type: redis`；但全部主代码 **0 处** `RedisTemplate/StringRedisTemplate/@Cacheable/@CacheEvict/@EnableCaching`（无 `@EnableCaching` 时 `spring.cache.type` 本身即空转）。

**影响**：部署要求 Redis 就绪（README/env.txt 均列为前置依赖），但应用实际不用它——运维成本与认知负担白付。

**整改建议**：**保留并落地用途**而非删除：v3 前置件 `AsyncJobService` 明确依赖 Redis（v3-proposals/00 §4）。建议在 AsyncJobService 落地时接通；若 v3 不立项再评估移除。

## C-06 文档目录三分与工具产物入库（P2）

**证据**：
- `doc/`（主文档索引，README/结构/模块/API/测试，维护良好）、`design-doc/`（v1.1/v2/v3-proposals 设计提案，职责清晰）、`docs/`（**仅 1 个文件** `superpowers/specs/2026-06-01-anti-slop-quality-gate-design.md`，未被 doc/README.md 索引，孤立）；
- `.codex/codebase/README.md` 被 git 追踪（`git ls-files` 确认），属本地工具产物；对照 `.serena/project.yml` 为有意共享的配置（合理）、`.codegraph/` 未入库（合理）；
- `frontend/doc/`（11 个文件）被 doc/README.md:20-22 正确索引，**不属于**三分问题。

**整改建议**：将 docs/ 下唯一文件移入 `design-doc/`（其内容即设计规格）并删除 docs/；`.codex/` 加入 .gitignore 并 `git rm --cached`。

## C-07 日志目录宿主机耦合（P2）

**证据**：`docker-compose.yml:14-15` 将日志/记录目录默认绑定到宿主机绝对路径 `/home/duwei/aienie-runtime/observability/data/ainovel/{logs,records}`；仓库根 `logs/` 目录为 root 属主（容器以 root 写入所致），内含 filebeat-data 等运行时产物。

**影响**：换机器/换用户部署需手工建目录并对齐权限；root 属主目录在仓库树内容易误触权限错误。

**整改建议**：默认值改为命名卷（named volume），宿主机路径仅作为显式覆盖项；仓库根 logs/ 确认由旧配置产生后清理。

## C-08 前端容器以 serve 提供静态文件（P2）

**证据**：`frontend/Dockerfile` 用 node:20-alpine + `serve@14` 起单线程静态服务（无 gzip、无缓存头）；生产流量依赖外层 Nginx（README 部署说明）兜底。

**整改建议**：镜像内改用 nginx:alpine 提供 dist（含 gzip/缓存头/SPA fallback），或明确文档化"该容器仅限内网、由外层 Nginx 终结"。优先级低。

## C-09 查询语义使用 POST（P2）

**证据**：`/v1/materials/review/pending` 为 POST + 空 body 的纯查询（后端 `MaterialController.java:83`，前端 `mock-api.ts:945`）。复核澄清：前后端路径**是**对齐的（探查阶段"前端调用已弃用路径"系误报），仅 HTTP 语义不当。

**整改建议**：改 GET；或保持现状并在 API 文档标注原因（如查询参数复杂）。

# 测试审计（T 系列）

## T-01 后端核心模块零覆盖（P1）

**证据**（40 个测试类的分布）：quality 13 个（覆盖最佳）、v2 6 个、style 3 个、auth/economy/integration/material 各 2 个、admin/ai/metrics/prompt/security/user/world 各 1 个；**零测试包**：
- `story`（StoryController/StoryService/OutlineService/ManuscriptController——**核心创作链路**）；
- `settings`（SettingsController/SettingsService）；
- `adminauth`（AdminAuthController 管理端登录）。

测试基建本身健康：H2 内存库隔离（application-test.yml），不依赖外部 MySQL。

**整改建议**：优先补 story 包的服务层测试（OutlineService 的 JSON 树变换、ManuscriptService 的生成编排是回归重灾区）；结合 A-03 拆分后按新边界补测更省力。v3 的评估体系（v3-proposals/06 §3）依赖生成链路的行为稳定，此项是其质量前提。

## T-02 前端页面组件零覆盖（P1）

**证据**：仅 6 个测试文件，全部为工具层（lib/__tests__ 的 sso、sso-callback、mock-api、shortcuts 4 个 + Settings/Admin 的 utils 2 个）；vitest 配置 `environment: "node"` 且 include 仅 `src/**/*.test.ts`（vite.config.ts:24-27）——**当前配置根本无法运行组件测试**（无 jsdom/tsx 包含）。ManuscriptWriter（F-01）2557 行零测试。

**整改建议**：vitest 增加 jsdom 环境与 `*.test.tsx` 包含；对 F-01 拆出的 hook（useWritingSession 等）先行补测——hook 测试性价比最高，不必追求页面级快照。

## T-03 测试资源 ddl-auto 不一致（P2）

**证据**：`backend/src/test/resources/application.yml:12` 为 `update`，`application-test.yml:9` 为 `create-drop`——两份测试配置策略不一致，且前者与 C-01 同样依赖隐式默认。

**整改建议**：统一为 create-drop（H2 场景语义正确），随 C-01 一并修。
