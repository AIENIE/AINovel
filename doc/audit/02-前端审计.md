# 前端审计（F 系列）

范围：`frontend/src`（130 个 TS/TSX 文件，约 21,700 行；React 18 + TypeScript + Vite + shadcn/ui）。

## F-01 ManuscriptWriter.tsx 巨型组件（P1）

**位置**：`frontend/src/pages/Workbench/tabs/ManuscriptWriter.tsx`（2557 行，实测确认）。

**证据**：单组件内含 **81 个 useState、31 个 useEffect**，承担至少 10 类职责：

| 职责块 | 行域 |
|--------|------|
| 编辑器接线（内容同步/脏标记/自动保存/字数） | 659-743 |
| 场景导航与多选 | 745-783、998-1014 |
| Slop 质量门禁（分析/风险分/重写任务） | 1165-1208 |
| 剧情质量诊断（趋势/修订生成与应用） | 1210-1286 |
| 导出系统（任务/模板/章节区间） | 1078-1104、1338-1387 |
| 版本/分支控制（diff/合并/回滚） | 921-942、1141-1157、1288-1336 |
| 布局持久化（localStorage + 服务端 600ms 防抖） | 336-376、600-657 |
| 快捷键（冲突检测/上下文分发） | 814-853 |
| 场景 Tab 管理（拖拽排序） | 756-767、1016-1024 |
| 写作会话追踪（30s 心跳） | 861-910 |

同目录另有 5 个 500-670 行的 tab（StoryConception 661、LorebookPanel 647、AnalysisDashboard 601、KnowledgeGraphTab 592、OutlineWorkbench 566）。

**影响**：任何工作台改动都要理解 2557 行上下文；81 个状态的联动无法单测（当前该文件零测试，见 T-02）；v3 G1"轻松创作体验"要重构工作台交互，此文件是最大障碍。

**违反原则**：单一职责、KISS。

**整改建议**：按上表职责块拆为 EditorPanel、QualityDashboard（slop+plot 合一）、VersionControl、ExportManager、SceneOutlinePanel 等子组件 + `useWritingSession`/`useLayoutPersistence` 等自定义 hook；先拆非渲染逻辑（hook 化）风险最低。应在 v3 G1 启动前完成（见 04 路线图）。

## F-02 生产 API 客户端命名为 mock-api.ts（P2）

**位置**：`frontend/src/lib/mock-api.ts`（1230 行）。

**证据**：该文件是**唯一**的生产 API 层（全 src 除 `pages/auth/SsoCallback.tsx:13` 的 SSO 回调外，无其他 fetch/axios 调用），被 20+ 个页面/组件导入；内部无任何 mock 行为。结构：三个请求封装 `requestJson/requestForm/requestVoid`（:68-108）+ 统一错误提取 `safeErrorMessage`（:110-121）+ 按资源分组的 30+ 端点组（adminAuth :534、user :550、ai :594、admin :622、stories :719、outlines :798、manuscripts :841、worlds :868、materials :927、prompts :965、v2 :992-1229）。

**影响**：纯命名债，但误导性极强——新人（或 AI 工具）会以为存在真实 API 层而绕过它。

**整改建议**：重命名为 `lib/api-client.ts`（一次 IDE 重构即可）；顺带把 1230 行按资源组拆为 `lib/api/` 目录多文件。

## F-04 死代码与页面命名倒置（P2）

**证据**：
- `pages/WorldBuilder/WorldBuilderPage.tsx`（158 行）**未被任何路由与 import 引用**（App.tsx 路由仅 `/worlds`→WorldManager、`/world-editor`→Worlds、`/worlds/create`→CreateWorld），为死文件；同目录 `components/WorldMetadataForm.tsx`、`WorldModuleEditor.tsx` 仍被 Worlds.tsx 使用，**不可**一并删除；
- 命名倒置：`Worlds.tsx`（复数名）实为**单个世界编辑器**，`WorldManager.tsx` 实为**列表页**——与常识相反。

复核澄清：NovelManager.tsx 与 Workbench/tabs/StoryManager.tsx 职责清晰（独立列表页 vs 工作台内嵌管理），不属于重复。

**整改建议**：删除 WorldBuilderPage.tsx；将 Worlds.tsx 重命名为 WorldEditor.tsx（路由 `/world-editor` 已是此语义）。

## F-05 无全局 401 处理（P1）

**位置**：`frontend/src/contexts/AuthContext.tsx`。

**证据**：
- 启动时校验 token（`:26-40`，`getProfile()` 失败且判定为鉴权错误则清 token）——这部分正确；
- 但**会话中途** token 过期无全局处理：页面各自 catch 后仅弹 toast，用户滞留在已登出状态的页面上；
- 鉴权错误判定为字符串匹配（`:17-20`）：

```ts
return error.message.includes("401") || error.message.includes("403");
```

`safeErrorMessage` 返回的是后端 message 字段，若后端消息本身含"401"字样会误判，后端消息不含状态码时则漏判——脆弱。

**整改建议**：在 `requestJson/requestForm/requestVoid`（mock-api.ts:68-108）统一拦截 `response.status === 401/403`：抛带状态码字段的类型化错误（如 `ApiError extends Error { status: number }`），并触发全局登出/跳转 `/login?next=...`。改动集中在一个文件。

## F-06 未使用依赖（P2）

**证据**（package.json 对照全 src grep）：
- `swagger-ui`、`swagger-ui-react`：0 处引用（node_modules 中连带引入 300+ 传递依赖）；
- `date-fns`：0 处引用（日期展示用原生 `toLocaleString`）；
- `next-themes`：仅 `components/ui/sonner.tsx:1` 引用 `useTheme`，项目无主题切换功能，属 shadcn 模板残留；
- `made-with-dyad.tsx`：**在用**（Index.tsx:5/115、AppLayout.tsx:27/162 渲染），为脚手架品牌标识，是否保留属产品决定，不算死代码。

**整改建议**：卸载 swagger-ui、swagger-ui-react、date-fns；将 sonner.tsx 的 theme 固定值化后卸载 next-themes。

## F-07 TypeScript 严格模式关闭（P2）

**位置**：`frontend/tsconfig.json:12-17`。

**证据**：`noImplicitAny: false`、`strictNullChecks: false`、`noUnusedLocals: false`、`noUnusedParameters: false`。实际代码违规率很低（`as any` 仅 2 处、无 @ts-ignore、非空断言仅 main.tsx:5 一处）——代码质量高于编译器要求，开严格模式的成本窗口正好很小。

**整改建议**：分两步：先开 `noImplicitAny + noUnusedLocals`（预计少量修复），再开 `strictNullChecks`（工作量集中在 mock-api.ts 归一化函数）。趁 F-02 拆文件时一并做。

## F-08 模型锁定三处重复表达（P2）

**证据**：`deepseek-v4-flash` 锁定同时存在于：
1. `frontend/src/lib/mock-api.ts:53`（`REQUIRED_AI_MODEL_KEY`）+ `:597/:606` 强制覆盖调用方传入的 modelId + `:57-66` 硬编码 FALLBACK_AI_MODELS；
2. 后端 `ai/AiModelPolicy.java`（策略锁定）；
3. `backend/src/main/resources/application.yml:61`（`AI_MODEL` 默认值）。

**影响**：这是当前有意的产品约束（v3-proposals/00 §6 明确"方案 A 必须在锁定模型下可行"），不算缺陷；但 v3 计划分三步解锁 AiModelPolicy——届时**三处**都要同步改，前端的强制覆盖（:597/:606）会静默吃掉后端已放开的模型选择。

**整改建议**：把"允许的模型"信源收敛到后端（`/v1/ai/models` 已存在）：前端移除强制覆盖与硬编码回退表，改为使用后端返回的默认模型标记。建议随 v3 模型解锁第一步一并做（见 04）。

## F-09 路由守卫重复（P2）

**证据**：`App.tsx:48-54`（ProtectedRoute）与 `:57-89`（AdminRoute）共享"loading 态渲染 + `next` 参数编码 + 重定向"骨架，约 25% 重复；AdminRoute 额外在**每次路由变化**时调 `api.adminAuth.me()` 远程校验（`:60-77`），与用户侧的本地缓存策略不一致。

**整改建议**：抽 `createGuardedRoute({ check, loginPath })` 工厂；AdminRoute 的每次导航远程校验可加会话级缓存（或接受现状，标注为有意的安全权衡）。
