# 整改路线图

编号引用见 `00-审计总览.md` 总清单，问题详情见 `01`~`03` 分域报告。工作量估计：**S** ≤ 0.5 人日、**M** 0.5~2 人日、**L** > 2 人日。

分阶段原则（与 `design-doc/v3-proposals` 对齐）：
- **阶段一**：v3 立项前必须清理——数据风险、以及 v3 四目标（G1-G4）将直接构建其上的模块的结构债；
- **阶段二**：与 v3 对应模块改造合并执行，避免二次返工；
- **阶段三**：机会性清理，顺手做、不单独排期。

## 与 v3 推进顺序的映射（v3-proposals/06 §1）

```
v3 推进：  [立项门槛]      阶段一(G2)        阶段二(G1)      阶段三(G3+G4摄入)   阶段四(G4)
本路线图：  #1-#9 + 快赢    #10(G2前置)      #8已完成 #11 #15   #13(AsyncJob已备)   （无新增欠账）
            #7 Flyway       #12(模型解锁)
            #5 V2清理(G4地基) #14(M1时)
```

核心结论：**阶段一的 9 项 + 快赢是 v3 立项门槛**（其中 #1 当日可完成；#5、#8 是工作量主体，合计约 1-2 周）；阶段二各项不单独排期，跟随 v3 对应目标立项时并入工作包；阶段三零排期压力。全部完成后，v3 四个目标将构建在无双实现死代码、有迁移机制、有测试兜底的地基上。
