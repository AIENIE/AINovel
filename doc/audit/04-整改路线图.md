# 整改路线图

编号引用见 `00-审计总览.md` 总清单，问题详情见 `01`~`03` 分域报告。工作量估计：**S** ≤ 0.5 人日、**M** 0.5~2 人日、**L** > 2 人日。

分阶段原则（与 `design-doc/v3-proposals` 对齐）：
- **阶段一**：v3 立项前必须清理——数据风险、以及 v3 四目标（G1-G4）将直接构建其上的模块的结构债；
- **阶段二**：与 v3 对应模块改造合并执行，避免二次返工；
- **阶段三**：机会性清理，顺手做、不单独排期。

这些项单独做会与 v3 改造重叠返工，应与对应 v3 工作项同批立项。

- **验收**：单类注入依赖 ≤ 6；G2 新增代码不进入 ManuscriptService 旧体。

### 11. 采纳 React Query 【F-03 | P1 | M-L | 挂靠 G1】

- **时机**：G1 前端产品化阶段（06 §2 的 P2 期）逐页面迁移 `useQuery/useMutation`；新页面即日起强制使用。
- **验收**：新增页面无手写 loading 样板；迁移页面切换往返不重复请求（缓存生效）。

### 12. 模型信源收敛 【F-08 | P2 | S-M | 挂靠 v3 模型解锁第一步】

- **时机**：v3 按 00 §6.2 解锁 AiModelPolicy 第一步（评审类任务放开备选模型）时同批执行：前端删除 `mock-api.ts:53/:597/:606` 的强制覆盖与硬编码回退表，模型可用性完全以 `/v1/ai/models` 为准。
- **验收**：后端调整模型策略后前端零改动即生效。

### 13. Redis 落地使用 【C-05 | P2 | 随 AsyncJobService】

- **时机**：v3 前置件 `AsyncJobService`（00 §4，依赖 Redis + MySQL）落地时自然消化；若 v3 最终不立项，则移除 starter 与配置。
- **验收**：Redis 出现真实调用方，或依赖被移除——不再存在"配置了但没人用"的中间态。

### 14. gRPC 客户端抽象 【A-08 | P2 | M | 挂靠 M1 多模型路由】

- **时机**：v3 M1 档（多模型路由）需要新增/改造外部 AI 服务客户端时，先抽 `GrpcEndpointManager`（端点解析 + 通道生命周期 + 元数据装配），Billing/AiGateway 两客户端随之瘦身。
- **验收**：第三个 gRPC 客户端出现时新增代码仅含业务方法。

### 15. 前端测试基建升级 【T-02 | P1 | M | 挂靠 G1】

- **时机**：与第 8 项（ManuscriptWriter 拆分）及 G1 前端改造同批：vitest 加 jsdom 环境与 `*.test.tsx` 包含，优先为拆出的 hook 补测。
- **验收**：`npm run test` 可运行组件/hook 测试；工作台核心 hook 覆盖率非零。
