## 审计方法

1. **探查阶段**：三路并行探查（后端架构 / 前端结构 / 配置·数据·测试），产出种子发现清单。
2. **复核阶段**：对每条种子发现逐项证实/证伪，要求精确到 `file:line` 的证据；**未经复核证实的结论一律不写入报告**。
3. **人工抽查**：对全部 P0 级与有争议的发现，由主审计通道直接读取源码二次确认。

复核阶段修正了以下探查误报（体现复核必要性）：

- **积分扣费并发安全**：初判为"读-改-写竞态（P0）"，复核证伪——`EconomyService.accountForUpdate()`（`backend/src/main/java/com/ainovel/app/economy/EconomyService.java:411`）调用的 `findForUpdateByUserId` 带 `@Lock(PESSIMISTIC_WRITE)`（`ProjectCreditAccountRepository.java:17`），实体另有 `@Version` 乐观锁兜底（`ProjectCreditAccount.java:34`）。**实现正确**。
- **`hooks/use-mobile.ts` 巨型文件**：初判 567 行，实际 18 行，实现规范。证伪。
- **NovelManager 与 Workbench/StoryManager 重复**：复核确认两者职责清晰（独立页面列表 vs 工作台内嵌单故事管理），**非重复**。
- **实体与表数量错配**：初判"50 实体 vs 49 表存在错配"，精确核对后为：49 张表、46 个 `@Entity` + 1 个 `@CollectionTable`(user_roles)，全部实体均有对应表；真实问题是 **2 张孤儿表**（见 C-04）。

## 做得好的方面（无需整改）

- **积分账本并发安全**：悲观锁 + 乐观锁双保险，幂等键（`findByUserAndIdempotencyKey`）齐备。
- **SlopQualityRecorder 接口双实现**：`InMemorySlopQualityRecorder`（测试用）与 `JpaSlopQualityRecorder` 通过策略接口分离——这是 A-01 的 V2 控制器**本应采用**的正确模式，仓库内已有范例。
- **前端类型集中定义**：`frontend/src/types/index.ts` 单点维护，无跨页面重复声明。
- **API 客户端集中**：除 SSO 回调外全部请求走统一封装（requestJson/requestForm/requestVoid），token 注入与错误提取一致。
- **shadcn/ui 使用一致**：无重复造轮子的自定义组件。
- **proto/gRPC 管理规范**：proto 为源码、生成物在 target/ 不入库。
- **git 卫生良好**：无构建产物入库、.gitignore 覆盖完整（除 C-06 的 `.codex` 一处）。
- **quality/ 包设计**（约 50 文件）：证据分级、门禁、漂移检测职责清晰，是 v3 G2 的可靠地基。
- **配置外部化**：除 C-01 一处外，全部配置支持环境变量覆盖，`@ConfigurationProperties` 用于复杂配置。

## 总体评估

代码库整体处于**功能推进快于结构收敛**的状态：核心领域（quality、economy、style、material）设计质量较高，但 v2 迭代期引入的"控制器内嵌内存实现 + 后补持久化服务"的过渡写法没有清理（A-01、A-05），前端工作台以单文件方式吸收了全部新功能（F-01），配置层存在一处真实的数据风险（C-01）。在 v3 提案（design-doc/v3-proposals，2026-07-05）立项前完成阶段一整改，可显著降低 v3 四个目标（G1-G4）的实施风险——具体对齐关系见 `04-整改路线图.md`。
