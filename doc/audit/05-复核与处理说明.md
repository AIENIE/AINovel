# 审计复核与处理说明

本文是对 `doc/audit/00`~`04` 的补充说明，不直接改写原审计快照。后续执行整改时，以当前代码为事实基线，以本文记录的复核意见修正原报告中的漂移表述、修法边界和优先级。

## 执行原则

1. 代码是事实来源，审计文档是待复核的假设集合；执行前必须重新确认目标文件和当前工作树状态。
2. 巨型类、巨型组件、超长函数必须作为正式修复项，不能降级为“单纯代码风格区别”。
3. 先处理真实风险和会阻碍后续整改的结构债，再处理命名、依赖、死文件、HTTP 语义等机会性清理。
4. 原审计中的历史快照信息保留，本文只提供执行校正，不要求同步重写 `01`~`04`。

## 认同并保留的审计点

- `C-01 + T-03`：`application.yml` 仍硬编码 `ddl-auto: update`，且默认数据源指向远程库；测试资源中也存在 `update` 与 `create-drop` 不一致。该项是真实数据风险，优先级最高。
- `A-06 + F-05`：后端异常边界和前端请求错误语义都不清晰。`RuntimeException` 统一映射 400 且无日志，前端只抛普通 `Error` 并用字符串匹配 401/403。
- `A-01 + A-05`：V2 控制器保留生产不可达的内存回退路径，同时返回 `Map<String,Object>`，导致生产路径、测试路径和 API 契约都分裂。
- `A-02 + A-10`：控制器直连 Repository、重复解析当前用户、事务标注在控制器上，这些都是分层问题。扩散面比原报告列举更大，应纳入 `AiController`、`MaterialController`、`WorldController` 等同型控制器。
- `A-03 + F-01`：`ManuscriptService`、`StoryService`、`ManuscriptWriter.tsx` 等巨型上下文必须拆分。`V2VersionController.mergeBranch(...)`、`StoryService.generateConceptionDraft(...)` 等长流程函数也应作为修复对象。
- `T-01 + T-02`：后端核心创作链路测试薄弱，前端测试基建尚不能支撑组件或 hook 测试，后续重构前必须补齐关键保护。

## 认同但需要调整修法

- `A-05` 不应一次性改完六个 V2 控制器。建议按 V2 垂直模块分批执行，每批同时删除内存回退、统一 DTO 契约、调整测试，不做只改 DTO 或只删分支的半截改动。
- `F-01` 原结论成立，但量化数字需校正。当前复核为 `2557` 行、`80` 个 `useState`、`30` 个 `useEffect`；原报告的 `81/31` 是小幅偏差，不影响“必须拆分”的结论。
- `F-03` 不宜写成全量迁移 React Query 的先决任务。更合适的规则是：新页面和被重构页面必须进入统一服务端状态模式，存量页面按触碰迁移。
- `F-08` 的问题不是简单重复配置，而是前端强制覆盖调用方传入的模型选择。后续应让模型策略以后端为准，前端只消费 `/v1/ai/models` 返回的可用模型。
- `F-09` 需要先确认 AdminRoute 每次导航远程校验是否属于有意安全策略，再决定是否抽象路由守卫工厂。
- `C-03` 是迁移治理专项，不是普通中等改动。引入 Flyway 或等价迁移框架前，必须确认生产库 baseline 策略、现有 `schema.sql` 可信度和部署流程配合。
- `C-05` 不应简单归为“删除 Redis”。当前代码中 `admin/ops/DependencyHealthService` 已通过 `RedisConnectionFactory` 执行依赖探活，因此它已不是“纯死配置删除”问题；后续应明确这是保留的运维探测依赖，还是继续演进为真实业务/基础设施能力，而不是沿用过时的“代码零使用”判断。
- `C-07` 是部署耦合问题，但 Aienie 运行时 observability 路径可能是跨仓库约定。不要在本项目里单点改成命名卷，除非同步处理运行时治理。

- `F-02`、`F-04`、`F-06`、`C-08`、`C-09` 均可保留为机会性清理，但不应抢占真实风险和结构整改的优先级。
